<objective>
Build the Phase-8 read foundation AND the first visible enrichment surface as one thin end-to-end slice: the edge `/api/lastfm/info` read proxy (LASTFM_KEY edge-only, absent-key graceful), the reusable `matchKey` normalization primitive, the optional `Track` enrich fields, the client `services/lastfm.ts` enrichment service, and the now-playing tag chips + higher-res cover swap that consume them.

After this plan a user who plays a track sees Last.fm top-tag chips under the title and (when strictly better) a higher-res album cover in the now-playing background — with zero impact when the track is not on Last.fm or no key is configured.

Purpose: Establish ENRICH-03's edge proxy + ENRICH-01/02's additive-merge-with-placeholder-filter on the highest-value surface (now-playing), and ship the match-key primitive Phase 13 reuses. Resolves both Claude's-discretion decision forks (route shape, data model) with rationale below.
Output: A working enriched now-playing view + the shared edge route, match-key, enrich service, and Track fields that Plans 02/03 build on.
</objective>

<phase_goal>
**As a** listener, **I want to** see Last.fm top tags and a higher-res cover on the track I'm playing, **so that** the now-playing view feels richer without ever slowing playback or losing the good source data.

</phase_goal>

<decision_forks_resolved>
**Fork 1 — Read-proxy route shape → DEDICATED `/api/lastfm/info` route (Candidate B).**
Rationale: the catch-all (Candidate A) forces widening `SourceId` to `'lastfm'`, which makes `SOURCES: Record<SourceId, SourceAdapter>` in `src/lib/sources/registry.ts` a type error until a Phase-10 client source exists (verified in 08-PATTERNS.md + registry typing). Candidate B mirrors the shipped `/api/similar` exactly, gives the absent-key-is-200-clean-empty posture for free (vs the catch-all surfacing Last.fm's own error JSON), and defers the `SourceId` change to Phase 10 where the client source actually lands. Key stays edge-only either way.

**Fork 2 — Enriched-fields data model → OPTIONAL fields on `Track`.**
Rationale: there is NO `serializeTrack` whitelist on the library persistence path — `library.svelte.ts` persists the full `Track` via `JSON.stringify`, so optional fields persist but are small + JSON-safe, and `Library.load()` reads them back untyped (no migration). The only field-dropping path is `history-logic.toEntry()`, which lists fields explicitly and will simply NOT carry the enrich fields (they re-enrich on replay — correct). A side cache adds a new store for marginal benefit. Optional `Track` fields is the lower-risk choice for Phase 8. Do NOT add enrich fields to `HistoryEntry`/`toEntry`.
</decision_forks_resolved>
