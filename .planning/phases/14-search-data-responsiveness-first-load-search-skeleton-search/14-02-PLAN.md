<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| module-level runes state ↔ SSR request | searchSession `$state` lives at module scope on the Cloudflare worker, shared across concurrent SSR requests |
| stale query partials → results | a superseded query's `onPartial` could write old rows into the live `results` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-05 | Information Disclosure | searchSession.svelte.ts module-level `$state` | mitigate | The page only WRITES the session inside browser-side handlers (onMount/onDestroy/run/loadMore); SSR never writes search state (mirrors settings/overlays/names guard discipline) — one user's session never bleeds into another's SSR render |
| T-14-06 | Tampering | stale onPartial after a new query | mitigate | Two-layer abort guard inside the page onPartial: `if (myAc.signal.aborted || kw !== q.trim()) return;` (captures the controller + query at run() start) — superseded partials never touch `results` |
| T-14-SC | Tampering | npm/pip/cargo installs | accept | Phase installs ZERO external packages (RESEARCH Package Legitimacy Audit: N/A) — no slopcheck needed |
</threat_model>
