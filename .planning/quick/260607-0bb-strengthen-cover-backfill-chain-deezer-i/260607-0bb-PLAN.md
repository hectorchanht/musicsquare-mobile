Fix the persistent "most home tiles are color blocks" symptom (this is the **4th attempt** at this exact problem — tests-green is NOT acceptance; a runtime human-verify is mandatory).

Keep Last.fm as the shelf data source (user explicitly decided NOT to switch to Deezer charts). Instead strengthen the cover BACKFILL so it actually covers the whole rendered page:

1. **Re-add iTunes** to the cover chain (restore the resolver wv8 deleted) — track chain becomes **Deezer → iTunes → CN → (Last.fm optional)**, artist chain becomes **Deezer → iTunes**.
2. **Lift the caps** so EVERY rendered tile + EVERY artist avatar is attempted, not just the first 24 — with bounded in-flight concurrency + per-call timeout + skip-cached so a warm visit costs ~0.
3. **Raise per-item success** via the multi-tier fall-through (a Deezer miss no longer strands a tile).

The current `max: 24` / `max: 12` means only ~24 tracks + ~12 artists are EVER attempted → every other tile stays a gradient forever. That cap, compounded by a single-source (Deezer-only) miss leaving a block, is the grounded root cause.

Output: restored `itunes-cover.ts` (+test), reworked `cover-backfill.ts` (+test) with the multi-tier chains and lifted caps, updated `+page.svelte` `scheduleBackfill` cap, and a passed runtime human-verify checkpoint.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → itunes.apple.com | Direct client fetch (CORS-open, no key); untrusted external JSON parsed for an artwork URL |
| browser → /api/deezer/search (edge proxy) → api.deezer.com | Existing proxy; Deezer CORS-blocked direct, so proxied; host-allow-listed to https *.dzcdn.net |
| browser → CN search proxies (via searchAll) | Existing pipeline; returns track metadata incl. cover URL |
| resolved cover URL → DOM `<img src>` | Untrusted URL string rendered as an image-source attribute |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-0bb-01 | Injection / Information Disclosure | resolved cover URL rendered into `<img src>` | mitigate | Client-side https-only guard before cache/notify (only a non-empty string starting with `https:` is cached/rendered); rendered as `<img src>` attribute ONLY (never CSS url()), so no CSS/script-injection surface; onerror→hide keeps a bad URL from showing a broken image. Same posture as caaReleaseGroupCover (T-nza-02). |
| T-0bb-02 | Denial of Service (self-DoS) | lifted total cap fanning out over ~270 tiles | mitigate | In-flight pool bounded to CAP=6 (mapWithConcurrency, not Promise.all); every resolver call bounded by AbortSignal.timeout(6000); skip-cached + de-dupe so a warm visit issues ~0 requests; Deezer is tier-1 and edge-cached (≤~50 req/5s); iTunes (soft-limited) + CN (fragile) fire ONLY on a Deezer miss, so the deep chain runs for a small minority of tiles. |
| T-0bb-03 | Information Disclosure | query terms crossing the iTunes/Deezer/CN boundaries | accept | Only already-public Last.fm artist/title text crosses the boundary; NO secret/key/PII; NO new env var (iTunes is keyless, Deezer proxy is public, CN is existing). |
| T-0bb-04 | Tampering | npm/pip/cargo installs | mitigate | NO new dependency added (plain fetch + URL + URLSearchParams); no install task → no package-legitimacy gate required. |
