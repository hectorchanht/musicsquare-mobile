# Quick Task 260606-6p7: App-language selector + whole-app i18n Summary

Added an App-language selector (English / 繁體中文 / 简体中文) to the settings page and internationalized all static UI chrome across the 12 in-scope `.svelte` files via a custom, dependency-free, Svelte-5-runes `t(key, params?)` that re-renders reactively on language change and persists to `openmusic:settings:v1`.

## What Was Built

- **`src/lib/i18n/` module** (no new dependency):
  - `index.ts` — `AppLang` type, `TranslationKey` (keyof en, so missing keys are compile errors at call sites), `dicts` registry, pure `interpolate()` (`{token}` substitution, unknown tokens left intact), pure `lookupKey()` (lang → en → raw key, never blank), pure `detectAppLang(navLang?)` (zh-TW/Hant/HK/MO → zh-Hant; other zh → zh-Hans; else en), and the reactive `t()` wrapper that reads `settings.appLang` on every call.
  - `en.ts` / `zh-Hant.ts` / `zh-Hans.ts` — dot-namespaced dictionaries with identical key sets. en values are the current verbatim on-screen text (en is a visual no-op); the two Chinese dicts are typed `Dict` so any missing/extra key is a compile error.
  - `i18n.test.ts` — plain node-project Vitest (imports only pure exports, never `t`): identical-key-set check, interpolation, en-then-raw-key fallback, `detectAppLang` mapping, and a no-blank-values guard. 11 tests.
- **`settings.appLang`** — `$state<AppLang>('en')`, mirrored on `lyricsLang`: load() honours a saved value, else first-visit auto-detect from `navigator.language` (handles both "stored settings without appLang" and "no stored settings at all"); save() persists it. `nameLang`/`lyricsLang` untouched and independent.
- **Settings page** — new App-language chip section (endonym labels rendered literally) wired to `settings.appLang` + `save()`; every settings-page string routed through `t()` (title, aria-labels, headings, option chrome, helper notes, data counts via param interpolation, flash/confirm). Proper-noun sources (NetEase/QQ/Kuwo/JOOX), quality tokens (320k/128k) and endonyms stay literal.
- **Remaining 8 files** — layout (nav via `labelKey` + `t(tab.labelKey)`, now-bar aria + loading), home, search (placeholder preserved verbatim), NowPlaying (transport/grip/lyrics aria, subnav, empty/status states), TrackMenu (menu labels, Like/Liked, detail modal, prompt, all 9 toasts), library, artist (pluralization via trackOne/trackMany + param lines), album — all chrome through `t()`; `names.dn` content and track-data values left untouched.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] svelte-check rejected `@ts-expect-error` directives in the i18n test**
- **Found during:** Task 2 (its `pnpm check` gate)
- **Issue:** Two `delete dicts[...][key]` lines and two unknown-key `lookupKey('does.not.exist', ...)` calls carried `@ts-expect-error`, but the `Dict` index signature / `TranslationKey | string` parameter mean these are NOT type errors, so svelte-check flagged the directives as "Unused" (2 errors).
- **Fix:** Removed the unnecessary directives; cast the deletion target to `Record<string, string>` so it stays intentional and type-clean.
- **Files modified:** src/lib/i18n/i18n.test.ts
- **Commit:** 15b7cf6

**2. [Rule 1 - Bug] Loop variables named `t` shadowed the `t()` translation helper**
- **Found during:** Tasks 5 and 6 (svelte-check "This expression is not callable. Type 'Track' has no call signatures.")
- **Issue:** Several `{#each ... as t}` loops (NowPlaying queue + related; library liked/playlist/downloads; artist hit-songs; album tracks) bound `t` to a `Track`, shadowing the imported `t()` so `t('key')` inside those loops failed to compile.
- **Fix:** Renamed every colliding loop variable from `t` to `track` and updated all in-body references.
- **Files modified:** NowPlaying.svelte, library/+page.svelte, artist/[name]/+page.svelte, album/[name]/+page.svelte
- **Commit:** 2f0f3a3 (NowPlaying), 66f871c (library/artist/album)

### Intentional plan-pattern over literal verify grep

- The plan's Task-4 verify grep was `t('nav.home')`, but the plan body explicitly asked the nav to store a translation KEY per tab and call `t(tab.labelKey)` so the bottom nav re-renders reactively. I implemented the body pattern (`labelKey: 'nav.home'` + `t(tab.labelKey)`); the literal `t('nav.home')` substring therefore does not appear, but the key is present and routed through `t()`. This follows the plan's stated intent over the simplified grep heuristic.

### Out of scope (left as-is)

- `src/routes/(app)/artist/[name]/+page.svelte` `<title>` is `{name} · MusicSquare` (old brand). The plan did not list the artist `<title>` among strings to extract (it lists the album `<title>` only), and it is not visible body chrome, so it was left untouched.
