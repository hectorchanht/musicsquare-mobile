<objective>
Deliver the public-`Music/` half of D-11 that plan 03 deferred: a **hand-written Kotlin Capacitor plugin** (local, in `android/` — NO npm/git dependency, avoiding the supply-chain risk T-999.1-07 that got the community `capacitor-mediastore` plugin rejected) that writes downloaded audio into the **public `Music/OpenMusic/`** collection via the **MediaStore API**, then wire `blob-store.ts` native `put`/`del` through it. This upgrades the plan-03 `// TODO(999.1-06)` app-private marker to the locked, resolved target: `public-music-mediastore` (resolved 2026-06-12).

The bridge inserts into `MediaStore.Audio.Media.EXTERNAL_CONTENT_URI` with `RELATIVE_PATH = "Music/OpenMusic/"` and the `IS_PENDING` write pattern on Android 10+ (API 29+); on API ≤28 it uses the legacy `WRITE_EXTERNAL_STORAGE` path (the permission is declared with `maxSdkVersion="28"`). RESEARCH Pitfall 2: `@capacitor/filesystem`'s `Directory.Documents`/`Directory.ExternalStorage` only reach `Music/` on Android 9 and older, and neither it nor `capacitor-blob-writer` expose MediaStore — hence the bespoke bridge.

Purpose: D-11's user-visible requirement — a downloaded song appears in `Music/` to file managers and other audio apps — while keeping offline in-app playback working. The user-visible outcome is LOCKED: download → visible in public `Music/` AND plays offline in-app AND survives backgrounding; an app-initiated delete removes the MediaStore entry it created.

Output: `MediaStoreSaverPlugin.kt`, its `MainActivity` registration, manifest permissions, a TS `media-store.ts` wrapper (`registerPlugin`), blob-store native put/del routed through it, and a device-test sign-off.
</objective>

MediaStore write pattern (RESEARCH Pitfall 2; developer.android.com scoped storage — API 29+):
- `val collection = MediaStore.Audio.Media.getContentUri(MediaStore.VOLUME_EXTERNAL_PRIMARY)` (API 29+) else `MediaStore.Audio.Media.EXTERNAL_CONTENT_URI`.
- ContentValues: `DISPLAY_NAME = fileName`, `MIME_TYPE` (infer from extension: mp3→audio/mpeg, flac→audio/flac, m4a/aac→audio/mp4, ogg→audio/ogg), `RELATIVE_PATH = "Music/OpenMusic/"` (API 29+ only), `IS_PENDING = 1` (API 29+).
- `val uri = resolver.insert(collection, values)` → `resolver.openOutputStream(uri).use { it.write(bytes) }` → set `IS_PENDING = 0` and `resolver.update(uri, values, null, null)` (API 29+).
- Delete: `resolver.delete(uri, null, null)` for the content URI the plugin returned.

Android version branch (RESEARCH State Inventory line 396, Pitfall 2):
- API 29+ (Android 10+, >95% install base): MediaStore RELATIVE_PATH + IS_PENDING; NO runtime permission needed to write your own MediaStore entries.
- API ≤28 (Android 9 and older): legacy `Environment.getExternalStoragePublicDirectory(DIRECTORY_MUSIC)` + `WRITE_EXTERNAL_STORAGE` (runtime-requested); the permission is declared with `android:maxSdkVersion="28"` so it is a no-op on modern Android.

Offline-playback split (planner's chosen approach — justify in SUMMARY if you deviate):
- `put`: write to public Music/ via the bridge AND keep/return a readable handle. Simplest robust split: the bridge writes to MediaStore (public, visible) and returns the `content://` URI; blob-store stores that URI in the localStorage metadata index and, for offline playback, `get(uid)` opens it back through the resolver (bridge `readFromMusic` returning base64/bytes) OR blob-store also keeps the existing app-private copy from plan 03 as the offline-read source. Choose ONE; the locked outcome is: visible in public Music/ AND plays offline in-app. Document the choice.
- `del`: remove the MediaStore entry the app created via `deleteFromMusic({ uri })` (and the app-private copy if that approach was chosen).

Manifest additions (this plan):
- API ≤28 legacy write: `<uses-permission android:name="android.permission.WRITE_EXTERNAL_STORAGE" android:maxSdkVersion="28" />`
- No `READ_MEDIA_AUDIO` needed to write/read your own MediaStore entries; do NOT request broad media-read scopes.
