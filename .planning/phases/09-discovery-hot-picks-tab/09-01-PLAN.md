<objective>
Build the Phase-9 data foundation as a thin vertical slice: a new edge `/api/lastfm/discovery` endpoint for Last.fm LIST methods (charts/tags/geo/artist top-albums) with Cloudflare Cache API edge-caching, an extension to the existing `/api/lastfm/info` so `album.getInfo` also returns its ordered tracklist, the client discovery list builders in `services/lastfm.ts`, and the load-bearing `resolveStub` transform that turns a Last.fm `{artist,title}` stub into a playable `Track` via the existing resolver.

Purpose: Every Phase-9 surface (home shelves in Plan 02, artist top-albums + album tracklist in Plan 03) consumes these contracts. Building the endpoint + builders + resolveStub first means Plans 02/03 receive the contracts directly and never explore the codebase. This plan also delivers DISCO-01 (top hits) and DISCO-04 (edge-cached, signed-out, no rate-limit failures) at the data level.

Output: `src/routes/api/lastfm/discovery/+server.ts` (+ test), extended `src/routes/api/lastfm/info/+server.ts` (+ test), extended `src/lib/services/lastfm.ts` (+ test), new `src/lib/services/discovery.ts` (+ test).

ENDPOINT FORK DECISION (Claude's discretion, resolved): Candidate B — a NEW `/api/lastfm/discovery` route owns the LIST methods, while the existing `/api/lastfm/info` keeps the single-entity getInfo contract and is only extended to surface the album tracklist (it already allow-lists `album.getinfo`). Rationale: charts/tags/geo/topalbums all share a `{ items: [...] }` list contract that does NOT generalize from the single-entity `LastfmInfo` unwrap; a dedicated route keeps the two response contracts separate, allows per-method Cache-Control TTLs, and gets a focused test. The Phase-8 edge posture is copied verbatim (key edge-only, absent-key 200 graceful, scoped CORS, fetchWithRetry, no-leak).

CACHING DECISION (Claude's discretion, resolved): The Cloudflare Cache API (`caches.default`) lives on the edge `/api/lastfm/discovery` endpoint (and the extended `album.getinfo` path of `/api/lastfm/info`), keyed by the upstream Last.fm URL. Per-method TTLs: charts ~3600s, tags ~21600s, artist.getTopAlbums/album.getInfo ~86400s. Every discovery response carries `Cache-Control: public, max-age=<ttl>` (all data is public, key-only — no user data, no `sk`). Client-side fan-out concurrency cap (4 in-flight) lives in the Plan-02 home builder, not here; resolveStub is strictly lazy/on-tap (one tap → one searchAll).
</objective>
