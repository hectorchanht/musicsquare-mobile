## Summary

This phase wraps the existing SvelteKit SPA in a Capacitor 8 Android shell, distributed as a signed APK via GitHub Releases. Five of the six work areas are low-to-medium risk and well-trodden: the dual-adapter build switch, `API_BASE` plumbing, CORS allowlist, Capacitor scaffold, and the GitHub Actions signed-APK pipeline all have documented, current patterns and the codebase is unusually well-positioned (no `+page.server.ts`, no prerender directives, a single `<audio>` element, a clean `blob-store.ts` seam, and an existing same-origin CORS allowlist in `src/lib/proxy/http.ts`).

**Two areas carry real risk** and must drive the plan's task structure:

1. **Background audio (HIGHEST RISK, D-04).** The Android System WebView **does not implement the MediaSession Web API** [CITED: github.com/jofr/capacitor-media-session]. This means the app's existing `navigator.mediaSession` wiring (`src/lib/services/media-session.ts`) is a **silent no-op on Android** — no system media notification, no lock-screen controls. A native plugin is **mandatory**, not optional. Worse: Android will sleep/kill the WebView (and pause `<audio>`) within minutes of backgrounding unless a **foreground service with `foregroundServiceType="mediaPlayback"`** is running [CITED: developer.android.com/about/versions/14/changes/fgs-types-required]. The leading Web-API-shaped bridge plugin, `@jofr/capacitor-media-session`, is **stale (v4.0.0, peer `@capacitor/core ^6.0.0`, last published 2024-08)** [VERIFIED: npm registry] and does not officially support Capacitor 8. This is the phase's defining technical decision and needs a dedicated spike + device test.

2. **Public Music folder writes (D-11).** `@capacitor/filesystem`'s `Directory.Documents` and `Directory.ExternalStorage` **only work on Android 9 (API 28) and older** [CITED: capacitorjs.com/docs/apis/filesystem]. On Android 10+ (which is the entire realistic install base — minSdk for Capacitor 8 is 24 but >95% of devices run 10+), writing to the public `Music/` collection requires the **MediaStore API**, which `@capacitor/filesystem` + `capacitor-blob-writer` **do not provide**. This makes the locked public-Music decision disproportionately painful relative to the "quick APK" goal — see the documented fallback and tradeoff in Pitfall 2.

**Primary recommendation:** Build the five low-risk areas first as a thin vertical slice (scaffold → dual build → CORS → API_BASE → signed-APK CI producing an installable, foreground-playing APK), then treat background-media-controls and public-Music as isolated, device-validated spikes. Pin Capacitor 8.4.0; do not assume Capacitor 7 (the prompt's assumption is stale — current is 8.4.0, released the day of this research).

### Pattern 1: Dual-adapter build switch (D-01)
**What:** `svelte.config.js` reads `process.env.BUILD_TARGET` and swaps the adapter.
**When to use:** Every build. Web CI sets nothing (default cloudflare); native build sets `BUILD_TARGET=native`.
**Example:**
```js
// svelte.config.js — Source: SvelteKit adapter-static docs (svelte.dev/docs/kit/adapter-static)
import cloudflare from '@sveltejs/adapter-cloudflare';
import staticAdapter from '@sveltejs/adapter-static';

const native = process.env.BUILD_TARGET === 'native';
const config = {
  compilerOptions: { /* ...runes unchanged... */ },
  kit: {
    adapter: native
      ? staticAdapter({ pages: 'build', assets: 'build', fallback: 'index.html', precompress: false, strict: false })
      : cloudflare()
  }
};
export default config;
```
- `fallback: 'index.html'` is acceptable here: there is **no prerendered homepage to conflict with** (verified — no `prerender = true` anywhere, no `+page.server.ts`). The SvelteKit docs prefer `200.html` to avoid that conflict, but Capacitor serves `index.html` as the WebView entry point, so `index.html` is the pragmatic choice. [CITED: svelte.dev/docs/kit/adapter-static]
- `strict: false` is needed because dynamic routes (`/album/[name]`, `/artist/[name]`) are not prerenderable and rely on the SPA fallback. Verify no route hard-requires SSR (none found).

### Pattern 2: SPA mode for the native build
**What:** Disable SSR for static output so every route is client-rendered behind the fallback.
**Example:**
```ts
// src/routes/+layout.ts (or guard on BUILD_TARGET so web keeps SSR)
export const ssr = false;   // native SPA build
export const prerender = false;
```
**Codebase note:** `src/routes/(app)/+page.ts` `load` reads only `url.searchParams` (the share-token OG path) — it is SPA-safe (no server `fetch`, no platform access). It will simply produce `og: null` in the native build (no crawlers inside an APK). No change required. [VERIFIED: codebase — read `src/routes/(app)/+page.ts`]

### Pattern 3: `API_BASE` single fetch seam (D-03)
**What:** One helper prepends `API_BASE` to relative `/api/...` paths; empty on web (same-origin), set to the Pages URL on native.
**Where the call sites are** (the centralization surface — **smaller than expected**):
- Source adapters build relative URL strings then call `fetch` directly: `src/lib/sources/{netease,qq,kuwo,joox,fivesing,jamendo}.ts` (e.g. `netease.ts:46` `/api/netease/search?...`, `:87` `/api/netease/url?...`, `:90` `/api/netease/lrc?...`).
- Services: `src/lib/services/lastfm.ts:61,193`, `translate.ts:30`, `deezer.ts` (path consts at `:28-33`), `similar.ts:33`, plus `cover-backfill.ts`, `discovery.ts`, `itunes-cover.ts`.
- A spike page: `src/routes/spike/+page.svelte:164` (dev-only, can ignore).
**Recommended approach:** Introduce `apiFetch(path, init)` in `src/lib/services/api-base.ts` that does `fetch((import.meta.env.VITE_API_BASE ?? '') + path, init)`. Migrate call sites to it. Because audio `<audio>.src` is also a relative `/api/netease/url` for Netease (`netease.ts:87`), **that URL must also be rewritten** — set `track.audioUrl` through the same base helper, or the `<audio>` element will request `https://localhost/api/...` which does not exist in the APK. This is an easy-to-miss correctness bug (see Pitfall 3).
**Example:**
```ts
// src/lib/services/api-base.ts — NEW
const BASE = import.meta.env.VITE_API_BASE ?? '';   // '' on web, 'https://openmusic.pages.dev' on native
export function apiUrl(path: string): string { return BASE + path; }
export function apiFetch(path: string, init?: RequestInit): Promise<Response> {
  return fetch(apiUrl(path), init);
}
```
Vite bakes `VITE_API_BASE` at build time (`import.meta.env`), so the native build's `.env` (or CI env) sets it; the web build leaves it unset. [CITED: vitejs.dev env-and-mode]

### Pattern 4: CORS for Capacitor origins on the existing app (D-02)
**What:** The existing same-origin CORS allowlist lives in `src/lib/proxy/http.ts` (`ALLOWED_ORIGIN_PATTERNS` / `corsHeaders(origin)`), used by the `/api/[source]` catch-all. The dedicated routes (`/api/lastfm/*`, `/api/translate`, `/api/deezer/*`, `/api/similar`) may NOT all run through it — **verify each route applies CORS** before relying on it.
**Recommended approach (discretion: hooks vs per-route):** A `src/hooks.server.ts` `handle` hook is the cleanest single seam — it can add the allowlisted CORS headers and answer `OPTIONS` preflight for **all** `/api/*` routes in one place, rather than editing each `+server.ts`. There is currently **no `hooks.server.ts`** in the repo (verified — `find` returned nothing), so this is a clean add.
**Allowlist additions** (append to the existing pattern list):
```ts
// add to ALLOWED_ORIGIN_PATTERNS (src/lib/proxy/http.ts) or the new hook
/^https:\/\/localhost$/,        // Capacitor Android default (server.androidScheme: 'https')
/^http:\/\/localhost$/,         // legacy http scheme
/^capacitor:\/\/localhost$/     // iOS / legacy capacitor scheme (harmless to allow now)
```
**Cloudflare Pages gotcha:** SvelteKit's `handle` hook runs on the Cloudflare Pages Function (workerd) — `OPTIONS` must return a `Response` with the CORS headers and **not** hit downstream logic. Also ensure `Vary: Origin` is set (already done in `corsHeaders`) so Cloudflare's edge cache does not serve a wrong-origin cached response. [CITED: existing `http.ts` already sets `Vary: Origin`]
**Default `server.androidScheme`** in Capacitor is `https`, yielding WebView origin **`https://localhost`** (NOT `capacitor://localhost`, which is the iOS default). So the must-allow origin for this Android-only slice is `https://localhost`. [CITED: capacitorjs.com/docs/config — androidScheme default https]

### Pattern 5: Native MediaSession bridge (D-05) — the conditional wiring
**What:** `media-session.ts` keeps its pure helpers (`buildArtwork`, `safePositionState`, `playbackStateFor`). The runes player store (`player.svelte.ts`) currently calls `navigator.mediaSession.*`. On native, those calls are no-ops (no Web MediaSession in WebView) and must be replaced by plugin calls behind a `Capacitor.isNativePlatform()` branch.
**Example shape (plugin API mirrors the Web API — that is the whole point of jofr's design):**
```ts
// player.svelte.ts (sketch) — Source: github.com/jofr/capacitor-media-session README
import { Capacitor } from '@capacitor/core';
// const MediaSession = Capacitor.isNativePlatform() ? NativeMediaSession : navigator.mediaSession;
// You MUST: (1) set metadata, (2) EXPLICITLY setPlaybackState('playing'), (3) set ALL action handlers —
// the notification only appears once playbackState is 'playing' AND handlers are registered. [CITED: jofr README]
```
**Critical:** The plugin "cannot detect audio playback in the WebView automatically — you wire the `<audio>` element manually, explicitly set playbackState to 'playing', and explicitly set play/pause action handlers, or the notification never appears." [CITED: github.com/jofr/capacitor-media-session README]

### Anti-Patterns to Avoid
- **Reloading the WebView on resume (violates D-06):** Do NOT call `invalidateAll()` or navigate on `App.resume`. Capacitor restores the existing WebView automatically; the danger is an `appUrlOpen`/lifecycle listener that re-navigates. Mirror the web rule (memory: `launch_handler: focus-existing`). The `attach()` `visibilitychange` flush in `player.svelte.ts:688+` is resume-safe (it persists position, does not reload) — keep it.
- **`Access-Control-Allow-Origin: *` for `/api/*`:** Forbidden — the proxy carries the JOOX token; `*` makes it an open CORS relay (existing anti-pattern in `http.ts` comments). Use the allowlist.
- **Native-audio plugin to "fix" background audio:** Rejected — moving audio out of the WebView breaks the entire `player.svelte.ts` engine (failover, prefetch, blob playback, lyrics sync) which is built around the single `<audio>` element.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Android foreground service + media notification | Custom Java/Kotlin Activity + Notification | A maintained Capacitor plugin (jofr bridge OR capawesome FGS) | FGS lifecycle, API-34 type rules, notification channels, action PendingIntents are a minefield; plugins encapsulate the `startForeground` typing |
| Large Blob → file write | base64 string passed through bridge | `capacitor-blob-writer` | `@capacitor/filesystem.writeFile` base64-encodes (33% bloat + memory spike); blob-writer streams binary [CITED: capacitor-blob-writer README] |
| APK signing in CI | Manual `apksigner` invocation | `r0adkll/sign-android-release` action OR gradle `signingConfig` from secrets | Handles keystore decode, zipalign, v1/v2/v3 signing schemes [CITED: github.com/marketplace/actions/sign-android-release] |
| SPA routing fallback | Custom 404 → index redirect | adapter-static `fallback: 'index.html'` | Official, handles client-side route resolution [CITED: svelte.dev/docs/kit/adapter-static] |
| Public Music folder write | Direct `File()` to `/storage/emulated/0/Music` | MediaStore API via a dedicated plugin (see Pitfall 2) | Scoped storage on Android 10+ forbids direct public-dir writes |

**Key insight:** The native surface area is small and every piece has a maintained plugin EXCEPT the exact combination this phase needs (Web-MediaSession-shaped bridge that keeps audio in the WebView, on Capacitor 8). That gap is the whole risk.

## Common Pitfalls

### Pitfall 1: Android WebView has no MediaSession API + kills backgrounded audio (THE phase risk, D-04)
**What goes wrong:** The APK plays audio fine in the foreground, but (a) shows no system media notification / lock-screen controls, and (b) audio pauses or the app is killed within ~3–4 minutes of screen-off. [CITED: github.com/jofr/capacitor-media-session/issues/11 — user reports Android 11 kills app after 3-4 min on background HTML5 audio]
**Why it happens:** Android System WebView does not implement `navigator.mediaSession`, so the existing web wiring is a no-op. And without a foreground service of type `mediaPlayback`, the OS suspends the WebView process on screen-off. On Android 14+ (targetSdk 36 for Cap 8), starting an FGS without a declared `foregroundServiceType` throws `MissingForegroundServiceTypeException`. [CITED: developer.android.com/about/versions/14/changes/fgs-types-required]
**How to avoid:**
1. Add to `AndroidManifest.xml`: `FOREGROUND_SERVICE` + `FOREGROUND_SERVICE_MEDIA_PLAYBACK` permissions, and `android:foregroundServiceType="mediaPlayback"` on the service.
2. Adopt a plugin that starts an FGS for active media sessions. **First choice to spike: `@jofr/capacitor-media-session`** (Web-API-shaped → minimal `media-session.ts` bridge changes) — but it is stale (peer Cap 6). The spike must answer: *does v4.0.0 install and run under Capacitor 8.4.0?* If not, **fall back to `@capawesome-team/capacitor-android-foreground-service` (Cap-8-current)** and build the media notification + actions manually, or fork/vendor jofr and bump its peer/native deps.
3. Explicitly `setPlaybackState('playing')` and register all action handlers (play/pause/next/prev) or the notification never shows.
**Warning signs:** No notification on play; audio stops shortly after locking the screen; `MissingForegroundServiceTypeException` in `adb logcat`.
**This pitfall should be its own plan/spike with a device-test gate before the rest of the phase is declared done.**

### Pitfall 2: Public Music folder is not writable via @capacitor/filesystem on Android 10+ (D-11)
**What goes wrong:** `Filesystem.writeFile({ directory: Directory.ExternalStorage, ... })` or `Directory.Documents` to reach `Music/` fails or is sandboxed on modern Android.
**Why it happens:** `Directory.Documents` and `Directory.ExternalStorage` are **only available on Android 9 (API 28) and older** in `@capacitor/filesystem`. [CITED: capacitorjs.com/docs/apis/filesystem] On Android 10+ (scoped storage), writing to the shared `Music/` collection requires the **MediaStore API**, which neither `@capacitor/filesystem` nor `capacitor-blob-writer` expose. [CITED: developer.android.com scoped storage; ionic-team/capacitor-filesystem issue #28 — open feature request for MediaStore support]
**How to avoid / TRADEOFF FLAG for planner:**
- **Locked decision is public Music (D-11).** Achieving it requires an additional MediaStore-capable plugin. A community option exists (`capacitor-mediastore`, repo `odion-cloud/capacitor-mediastore`) **but it is NOT on the npm registry** [VERIFIED: `npm view capacitor-mediastore` → not found], so it would be a git/source dependency — unverified, single-maintainer, no slopcheck coverage. That is a meaningful supply-chain + maintenance risk for a "quick APK."
- **Documented fallback (evidence-backed):** Write downloads to the **app-private external dir** (`Directory.Data` / `Directory.External`, app-scoped, no MediaStore, no permissions on any Android version). Files survive until uninstall, are NOT user-visible in other apps, and `capacitor-blob-writer` + `@capacitor/filesystem` handle them with zero extra plugins. This is the **shortest path to a working download feature** and aligns with the "quick APK" bias.
- **Recommendation for discuss/plan:** Surface this as a decision point — public-Music (D-11) costs an unverified MediaStore plugin + per-version permission handling; app-private costs nothing and ships now. The locked decision says public Music, but the prompt explicitly authorized documenting the fallback "if disproportionately painful." It is. Planner should flag for user re-confirmation.
- **On uninstall:** app-private files are deleted; MediaStore-owned public files in `Music/` typically persist after uninstall (orphaned) — a UX consideration either way.

### Pitfall 3: Native `<audio>.src` pointing at a relative `/api/...` 404s in the APK (D-03 correctness)
**What goes wrong:** Netease audio plays on web but is silent/errors in the APK.
**Why it happens:** `netease.ts:87` sets `track.audioUrl = '/api/netease/url?id=...'` — a **relative** URL. In the WebView (origin `https://localhost`), that resolves to `https://localhost/api/netease/url` which does not exist (no server in the APK). All other sources set `audioUrl` to an absolute CDN URL, so only Netease (and any other route-resolved audio) is affected.
**How to avoid:** Route audio URL construction through the same `apiUrl()` helper as metadata (Pattern 3). Add a test asserting `track.audioUrl` is absolute when `VITE_API_BASE` is set.
**Warning signs:** Most sources play; one source is silent only in the APK.

### Pitfall 4: targetSdk / build-tool version mismatch for Capacitor 8
**What goes wrong:** Gradle build fails in CI or locally.
**Why it happens:** Capacitor 8 requires specific toolchain versions.
**How to avoid:** minSdkVersion=24, compileSdkVersion=36, targetSdkVersion=36; Gradle wrapper 8.14.3, AGP 8.13.0, Kotlin 2.2.20, **Node 22+**, and a JDK that satisfies AGP 8.13 (JDK 17 minimum; JDK 21 recommended/used by current Android Studio Otter 2025.2.1). [CITED: capacitorjs.com/docs/updating/8-0; capawesome.io upgrade-to-8 guide] CI must set up JDK (e.g. `actions/setup-java` Temurin 21) and Node 22. The repo already pins `engines.node >=22`.
**Warning signs:** `MissingForegroundServiceTypeException` only at runtime means targetSdk is 34+ (expected); a Gradle "unsupported class file version" means JDK too old.

### Pitfall 5: Versioning for Obtainium (D-08)
**What goes wrong:** Obtainium does not detect updates, or installs fail with "downgrade detected."
**Why it happens:** `versionCode` must monotonically increase across releases; Obtainium matches GitHub Release tags (prefers semver) and APK assets by name.
**How to avoid:** Derive `versionName` from the git tag (e.g. `v1.0.0` → `1.0.0`) and `versionCode` from a monotonic source (e.g. `github.run_number`, or a date-based integer like `YYYYMMDDNN`). Name the APK asset consistently (e.g. `musicsquare-v1.0.0.apk`). Tag with semver. [CITED: khromov.se Capacitor GitHub Actions guide]
**Warning signs:** Obtainium shows "up to date" after a new release; reinstall over an existing APK fails.

```yaml
# .github/workflows/android-release.yml — Source: khromov.se + marketplace/sign-android-release
on:
  push:
    tags: ['v*']
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22 }
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: 21 }
      - run: corepack enable && pnpm install --frozen-lockfile
      - run: BUILD_TARGET=native VITE_API_BASE=https://openmusic.pages.dev pnpm build
      - run: npx cap sync android
      - name: Set versionCode/Name from tag
        run: |   # inject into android/app/build.gradle (or use a gradle property)
          echo "VERSION_NAME=${GITHUB_REF_NAME#v}" >> $GITHUB_ENV
          echo "VERSION_CODE=${GITHUB_RUN_NUMBER}" >> $GITHUB_ENV
      - run: cd android && ./gradlew assembleRelease
      - uses: r0adkll/sign-android-release@v1
        with:
          releaseDirectory: android/app/build/outputs/apk/release
          signingKeyBase64: ${{ secrets.RELEASE_KEYSTORE }}
          alias: ${{ secrets.KEY_ALIAS }}
          keyStorePassword: ${{ secrets.KEYSTORE_PASSWORD }}
          keyPassword: ${{ secrets.KEY_PASSWORD }}
      - uses: softprops/action-gh-release@v2
        with:
          files: android/app/build/outputs/apk/release/*-signed.apk
```
(Alternatively inject `signingConfig` into `build.gradle` from secrets and let gradle sign during `assembleRelease` — one fewer action.)
