# Phase 999.1 Plan 05: GitHub Actions Signed-APK Release Pipeline Summary

Stood up the tag-triggered Android release CI (D-07/D-08/D-09): a `v*` tag push on the origin
repo builds the native SvelteKit SPA, syncs the Capacitor Android project, assembles a release
APK with **tag-derived versionName + monotonic versionCode** (Pitfall 5 — Obtainium never
downgrade-blocks), signs it with the consistent release keystore from masked repo secrets
(Security V6 — never echoed, scoped to the tag trigger), and publishes the signed APK as a
GitHub Release asset for Obtainium-based updates. **The plan is `autonomous: false`: Task 1
(generate keystore + write 4 GitHub Actions secrets) is a blocking human-action gate and Task 3
(end-to-end tag → signed Release → Obtainium dry-run) is a blocking human-verify gate — both
remain outstanding and are returned as a checkpoint.** Task 2 (the workflow + gradle versioning)
is complete and committed.

## What Was Built

**Task 1 — Generate release keystore + GitHub Actions secrets (BLOCKING human-action — NOT done)**
- Truly human-only: the private signing key must never leave the user's control / be committed,
  and Claude cannot write repo secrets. This is the Obtainium update trust anchor — generated
  once, backed up, reused for every release. See the checkpoint below for the exact steps.
- Pre-confirmed on the code side: `.gitignore` already ignores `*.jks` / `*.keystore` / `*.apk` /
  `*.aab` (Plan 02), so the keystore can never be accidentally committed (V6 / T-999.1-14).

**Task 2 — Tag-triggered build+sign+publish workflow + tag-derived versioning (D-07/D-08/D-09)** [commit af609e4]
- `.github/workflows/android-release.yml`, triggered on `push: tags: ['v*']`, `runs-on: ubuntu-latest`:
  `actions/checkout@v4` → `actions/setup-node@v4` (node 22) → `actions/setup-java@v4` (temurin 21)
  → `corepack enable && pnpm install --frozen-lockfile` → `BUILD_TARGET=native
  VITE_API_BASE=https://openmusic.lol pnpm build` → `npx cap sync android` → export
  `VERSION_NAME=${GITHUB_REF_NAME#v}` + `VERSION_CODE=${GITHUB_RUN_NUMBER}` to `$GITHUB_ENV` →
  `cd android && ./gradlew assembleRelease -PversionName=... -PversionCode=...` →
  `r0adkll/sign-android-release@v1` (releaseDirectory `android/app/build/outputs/apk/release`,
  signingKeyBase64 `${{ secrets.RELEASE_KEYSTORE }}`, alias/keyStorePassword/keyPassword from
  the other three secrets) → `softprops/action-gh-release@v2` (`files: .../release/*-signed.apk`,
  `fail_on_unmatched_files: true`). `permissions: contents: write` at workflow scope for the
  release upload.
- `android/app/build.gradle` `defaultConfig`: `versionCode` reads
  `project.hasProperty('versionCode') ? project.property('versionCode').toInteger() : 1` and
  `versionName` reads the `versionName` property with a `"1.0"` fallback — so `-PversionCode`/
  `-PversionName` from CI drive them (Pitfall 5) while local builds stay valid. `minSdk 24` /
  `targetSdk 36` (from `variables.gradle`) unchanged; `namespace`/`applicationId`
  `com.openmusic.app` (D-12) unchanged.
- No secret is echoed and no step enables shell tracing (`set -x`) — relies on GitHub's automatic
  `***` masking. The only `echo` statements write the public tag name + run number to
  `$GITHUB_ENV` (no secrets).

**Task 3 — End-to-end release dry-run (BLOCKING human-verify — NOT done)**
- Exercises the full pipeline against the origin repo with a test tag, then verifies the signed
  APK installs and Obtainium detects the release + a version bump without a downgrade error.
  Requires pushing a tag to origin (push is orchestrator/user-owned), a physical device, and
  Obtainium — none automatable here. See the checkpoint below.

## Deviations from Plan

### [Rule 1 — Correctness] Native API base is openmusic.lol, not the plan-text pages.dev
- **Found during:** Task 2 (reading `build:native` in package.json + Plan 02 SUMMARY).
- **Issue:** The plan text repeats `VITE_API_BASE=https://openmusic.pages.dev`, but Plan 02 (a
  later-executed decision) repointed the canonical native base to `https://openmusic.lol` — the
  custom domain on the same Pages project, device-verified to serve the Plan-01 CORS hook and echo
  `access-control-allow-origin: https://localhost`. A release APK baked against pages.dev would
  diverge from the verified, shipped debug APK.
- **Fix:** The workflow's build step uses `VITE_API_BASE=https://openmusic.lol`. Same trust
  boundary (same Pages project, same CORS allowlist) — no new threat surface.
- **Files:** .github/workflows/android-release.yml — **Commit:** af609e4

### [Rule 2 — Missing functionality] Release-upload permission + fail-fast on missing APK
- **Found during:** Task 2 (action-gh-release needs write to create the Release).
- **Issue:** The bare RESEARCH skeleton omits `permissions:`; on repos where the default
  `GITHUB_TOKEN` is read-only, `action-gh-release` cannot create the Release. Also, a silently
  failed sign step would publish an empty Release.
- **Fix:** Added `permissions: contents: write` at workflow scope and `fail_on_unmatched_files: true`
  on the upload (no `*-signed.apk` → the job fails loudly). Pinned `BUILD_TOOLS_VERSION: 34.0.0`
  on the sign step so zipalign/apksigner resolve deterministically.
- **Files:** .github/workflows/android-release.yml — **Commit:** af609e4

No bugs (Rule 1 beyond the URL) in the workflow. No architectural (Rule 4) changes. No auth gates
hit during Task 2 (the `gh`/secrets work is the explicit Task 1 human-action gate, not an
incidental gate).

## Threat Surface

Conforms to the plan's threat model; no new surface.
- **T-999.1-14 (keystore/password leakage in CI logs):** mitigated — keystore base64 + 3 passwords
  are `secrets.*` inputs only (auto-masked `***`); no `echo`/`set -x` on the sign step (audited
  empty); job scoped to the `v*` tag trigger; keystore gitignored (`*.jks`/`*.keystore`).
- **T-999.1-15 (inconsistent/lost signing key breaks update trust):** mitigated by design — a single
  consistent keystore is generated once in Task 1 and reused via the `RELEASE_KEYSTORE` secret every
  release. Backup is part of the Task 1 gate instructions.
- **T-999.1-16 (downgrade/replay via non-monotonic versionCode):** mitigated — `versionCode` =
  `GITHUB_RUN_NUMBER` (monotonic), `versionName` = `${tag#v}` (semver); the Task 3 dry-run is the
  Obtainium update-detection gate.
- **T-999.1-17 (malicious pinned action versions):** accepted — actions pinned to major tags
  (`@v4`/`@v1`/`@v2`) from known publishers; SHA-pinning is the documented out-of-scope hardening.
