<objective>
Prefetch the NEXT track while the current song plays, so it starts instantly on track-end or next-tap.

The latency a user feels on next/ended is the per-source proxy round-trip inside `ensureTrackDetails` (resolve audioUrl + lyrics), NOT byte buffering. Because `ensureTrackDetails` is idempotent (its readiness guard short-circuits a `detailsLoaded` track) and `play()` syncs the resolved track back into `queue[i]`, pre-resolving the next track means the later `play()` hits a no-op resolve = instant start.

Purpose: Near-instant next/auto-advance with zero new audio elements, mirroring the real `next()` selection so prefetch matches what actually plays.
Output: A private `prefetchNext()` on the `Player` store, fired best-effort from `play()`, plus node unit tests.
</objective>

    - Stale-guard: if `current` changed during the resolve so the just-prefetched track is no longer the "next" of the current track, the resolved result is discarded (not written into queue) and no error surfaces. Use AbortController + a generation/uid check; ensureTrackDetails accepts a signal.

    `prefetchNext()` logic, mirroring next()'s REAL selection: let `i = this.indexOf(this.current)`; if `i < 0` return. If valid, `this.queue[slot] = resolved` (same in-place sync play() does). Otherwise discard silently (do NOT write — avoids clobbering a newer prefetch / stale write). The whole body is wrapped so it NEVER throws (catch and ignore — best-effort, matching ensureAhead/regenerate).
