<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Generate release keystore + add GitHub Actions secrets (signing trust anchor)</name>
  <what-built>Nothing automated — this is the one truly human-only step: generating the private signing key (must never leave the user's control / be committed) and writing repo secrets, which Claude cannot do.</what-built>
  <how-to-verify>
1. Generate a release keystore locally:
   `keytool -genkeypair -v -keystore release.jks -alias openmusic -keyalg RSA -keysize 2048 -validity 10000`
   (choose and SECURELY RECORD the keystore password, key alias `openmusic`, and key password — these are reused for EVERY future release; losing them breaks Obtainium update continuity and forces all users to reinstall).
2. Base64-encode the keystore: `base64 -i release.jks | pbcopy` (macOS) — this is the RELEASE_KEYSTORE value.
3. On the ORIGIN GitHub repo → Settings → Secrets and variables → Actions, add four repository secrets:
   - `RELEASE_KEYSTORE` = the base64 string from step 2
   - `KEY_ALIAS` = openmusic
   - `KEYSTORE_PASSWORD` = the keystore password
   - `KEY_PASSWORD` = the key password
4. Back up `release.jks` + the passwords in a secure location (password manager). Do NOT commit the keystore (Plan 02 gitignored *.jks/*.keystore — confirm).
5. Confirm `git status` shows no keystore staged.
  </how-to-verify>
  <resume-signal>Type "secrets-set" once the four Actions secrets exist on the origin repo and the keystore is backed up + NOT committed. Confirm the alias you used (default expected: openmusic) so Task 2's build.gradle/signing matches.</resume-signal>
</task>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner ↔ repo secrets | The base64 keystore + passwords are decoded in CI to sign the APK |
| signed APK → end users (sideload/Obtainium) | The signing key is the only integrity/trust anchor for sideloaded updates |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-14 | Information Disclosure | keystore/password leakage in CI logs | mitigate | Base64 keystore + passwords in GitHub Secrets (auto-masked); NO echo/shell-trace on the signing step; job scoped to the v* tag trigger; keystore never committed (gitignored) — Security V6 |
| T-999.1-15 | Spoofing/Tampering | inconsistent or lost signing key breaks update trust | mitigate | Single consistent keystore generated once (Task 1), backed up, reused every release = the Obtainium trust anchor; documented in user_setup |
| T-999.1-16 | Tampering | downgrade/replay via non-monotonic versionCode | mitigate | versionCode from GITHUB_RUN_NUMBER (monotonic); versionName from the semver tag; Task 3 dry-run confirms Obtainium update detection without downgrade errors (Pitfall 5) |
| T-999.1-17 | Tampering | malicious pinned action versions | accept | Actions pinned to major tags (checkout@v4, setup-node@v4, sign-android-release@v1, action-gh-release@v2) from known publishers; SHA-pinning is a hardening option but out of scope for this quick-APK slice |
</threat_model>
