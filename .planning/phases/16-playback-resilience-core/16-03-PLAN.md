<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| player.notice (store) → layout toast host (UI) | Store-originated messages cross into the UI; the host must only read + invoke the provided action, never mutate store internals. |
| i18n token interpolation | `{title}`/`{count}` values flow from track data into rendered toast strings. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-07 | Information Disclosure / Injection | toast message interpolation of `{title}` | accept | Svelte escapes interpolated text by default (no `{@html}`); a hostile track title cannot inject markup into the toast. Titles are display data already shown across the app (nowbar/now-playing) — no new exposure. |
| T-16-08 | Tampering | Retry button invoking notice.action | mitigate | The host only calls the store-provided `notice.action` closure (defined in 16-02 as `recoverFromStop` → reset counter + next()); the UI cannot construct an arbitrary action, so the recovery path is fixed by the store, not the UI. |
| T-16-09 | Denial of Service | toast storm flooding the UI | mitigate | Skip notices are batched in 16-02 (burst-collapse window) and the host replaces rather than stacks (single `notice` channel + cleared timer); the loop-guard stop halts further skips after the cap, so toasts cannot flicker faster than the batch window. |
</threat_model>
