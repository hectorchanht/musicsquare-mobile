# Phase 999.1: Code Review Report

**Reviewed:** 2026-06-12T04:29:02Z
**Depth:** standard
**Files Reviewed:** 32
**Status:** issues_found

## Summary

Reviewed the Capacitor native-shell migration: API_BASE fetch seam + central CORS hook, the hand-written Kotlin MediaStore plugin and its TS bridge, the platform-switched blob store, the native media-session adapter, the dual-adapter player-store wiring, and the tag-triggered signed-release workflow.

The security fundamentals asked about in the review brief are largely sound: the CORS allowlist never emits `*` and never echoes unknown origins (verified in `http.ts` and pinned by `proxy.test.ts`); no keystore or keystore password is committed (`git ls-files` confirms only `gradle-wrapper.jar` as a binary under `android/`); the release workflow passes secrets only as action inputs with no echo or shell tracing; the JOOX token appears nowhere in client code.

However, three Critical defects were found. (1) The central CORS hook — whose stated purpose is to make `/api/translate` reachable from the native WebView — advertises `Access-Control-Allow-Methods: GET, OPTIONS` only, so the preflighted POST to `/api/translate` fails CORS from the APK: the one route the hook was built for does not work cross-origin. (2) The player constructs `new MediaMetadata(...)` unguarded on the native path, but in Chromium the `MediaMetadata` constructor is gated by the same MediaSession feature as `navigator.mediaSession`, which the adapter's own header documents as absent in the Android System WebView — if that premise holds, `play()` throws a `ReferenceError` before `audio.src` is ever assigned and native playback fails entirely. (3) The public-Music write path requires the `WRITE_EXTERNAL_STORAGE` runtime permission on API 24–28 (minSdk is 24) but never requests it, so the D-11 public `Music/OpenMusic/` feature is silently dead on Android 7–9 and drags `put()` to `false` despite a successful app-private write.

Warnings cluster around the blob-store's base64 round-trips (an OOM/ANR risk the module's own header explains how to avoid), the Kotlin plugin's IS_PENDING/path-handling edges, release-pipeline robustness (versionCode reset hazard, archived signing action on a mutable tag), and a failure-pinning cache inconsistency in `lastfm.ts`/`similar.ts`.

## Critical Issues

### CR-01: CORS hook does not allow POST — `/api/translate` is broken from the native app (the exact case the hook exists for)

**File:** `src/lib/proxy/http.ts:31` (consumed by `src/hooks.server.ts:27,34`)
**Issue:** `corsHeaders()` hardcodes `'Access-Control-Allow-Methods': 'GET, OPTIONS'`. `/api/translate` is the only POST route under `/api/` (`src/routes/api/translate/+server.ts:38`), and the client sends it with `content-type: application/json` (`src/lib/services/translate.ts:31-35`), which forces a CORS preflight. From the native WebView origin `https://localhost`, the preflight asks `Access-Control-Request-Method: POST`; the 204 response lists only `GET, OPTIONS`, so the browser rejects the preflight and the POST never fires. `translateLines()` swallows the failure and returns the untranslated lines, so lyric translation silently never works in the APK. This directly defeats the hook's documented purpose (`hooks.server.ts:4-7`: "/api/translate ... would reject the native WebView origin ... Lifting CORS into one handle() covers translate"). The test suite (`src/routes/api/proxy.test.ts:126-184`) never asserts the allow-methods value or exercises a POST preflight, which is why this passed.
**Fix:**
```ts
// src/lib/proxy/http.ts
'Access-Control-Allow-Methods': 'GET, POST, OPTIONS',
```
And add a proxy.test.ts case asserting a preflight for POST `/api/translate` from `https://localhost` returns an `Access-Control-Allow-Methods` header containing `POST`.

### CR-02: Unguarded `new MediaMetadata(...)` on the native path — likely `ReferenceError` that aborts `play()` before `audio.src` is set

**File:** `src/lib/stores/player.svelte.ts:1335, 1400, 1493` (premise documented in `src/lib/services/native-media-session.ts:3-4`)
**Issue:** The native adapter exists because (per its own header) "The Android System WebView has NO `navigator.mediaSession`". In Chromium, the `MediaMetadata` constructor is exposed behind the same MediaSession feature gate as `navigator.mediaSession` — where one is absent, both are. Yet `play()` constructs `new MediaMetadata({...})` unconditionally on the native branch (the `ms` accessor returns the native adapter, so the `if (ms)` guard passes) at lines 1335 (offline-blob path) and 1400 (network path), both BEFORE `this.audio.src` is assigned. If `MediaMetadata` is undefined in the WebView, the `ReferenceError` is caught by `play()`'s outer `catch` (line 1462), `this.error` is set, and playback never starts — total native playback failure, on every track. `resolveCoverAsync` (line 1493) has the same construction. The device-matrix UAT for this phase is still pending (checkpoint persisted as pending human UAT), so this has not been disproven on hardware.
**Fix:** The native adapter only reads `.title/.artist/.album/.artwork` from the value (`native-media-session.ts:64-69`), so a structural literal suffices. Add a guarded factory and use it at all three sites:
```ts
function makeMetadata(init: MediaMetadataInit): MediaMetadata {
	return typeof MediaMetadata !== 'undefined'
		? new MediaMetadata(init)
		: ({ ...init } as unknown as MediaMetadata);
}
// ...
ms.metadata = makeMetadata({ title: ..., artist: ..., album: ..., artwork: buildArtwork(...) });
```
Verify on a physical device either way; if the WebView does expose `MediaMetadata`, keep the guard as cheap insurance against the documented premise.

### CR-03: Public-Music write needs runtime `WRITE_EXTERNAL_STORAGE` on API 24–28 but never requests it — D-11 feature dead on Android 7–9, and it drags `put()` to `false`

**File:** `android/app/src/main/java/com/openmusic/app/MediaStoreSaverPlugin.kt:92-109` (with `AndroidManifest.xml:59-61`, `android/variables.gradle:2` minSdk 24, `src/lib/services/blob-store.ts:101-113`)
**Issue:** `WRITE_EXTERNAL_STORAGE` is a *dangerous* (runtime) permission from API 23. The manifest declares it with `maxSdkVersion="28"`, but neither the plugin (`@CapacitorPlugin` has no `permissions` block, no `requestPermissions` call) nor any TS code ever requests the runtime grant. On API 24–28 devices (minSdk is 24, so these are supported), `outFile.outputStream()` throws `FileNotFoundException (EACCES)` on every call → `call.reject(...)` → `saveToMusic` always fails. The manifest comment ("only the legacy API <=28 branch needs WRITE_EXTERNAL_STORAGE ... T-999.1-18 mitigation") declares the install-time permission but the mitigation is incomplete without the runtime request, so the public `Music/OpenMusic/` deliverable silently never works on Android 7–9. Compounding it, `nativePut` wraps the private write AND the public write in one try/catch, so the rejection makes `put()` return `false` even though the app-private offline copy (step 1) landed and is fully readable by `nativeGet` — the never-throws sentinel now reports failure for a download that actually works offline.
**Fix:** Either (a) request the permission at call time on the legacy branch — annotate the plugin with `@CapacitorPlugin(name = "MediaStoreSaver", permissions = [Permission(strings = [Manifest.permission.WRITE_EXTERNAL_STORAGE], alias = "publicMusic")])`, check `getPermissionState` and call `requestPermissionForAlias` before the API<=28 write — or (b) explicitly drop public-Music support below API 29 (return `call.resolve()` with no uri on `SDK_INT < Q`) and document it. In both cases also apply the WR-01 fix so a public-copy failure cannot fail the whole `put()`.

## Warnings

### WR-01: `nativePut` conflates public-copy failure with total failure — violates its own best-effort contract

**File:** `src/lib/services/blob-store.ts:101-113`
**Issue:** The module's posture doc says "a failed public-Music write degrades to CDN playback" (best-effort), and `put()`'s contract is "Returns true if the write landed." But step 1 (private copy via `write_blob`) and step 2 (public copy via `saveToMusic`) share one try/catch: any step-2 failure (CR-03's permission case, a transient MediaStore error, an OOM in `blobToBase64`) returns `false` even though the offline-read source — the thing `get()` actually serves playback from — landed. Callers and tests treat `false` as "nothing persisted."
**Fix:** Make step 2 independently best-effort:
```ts
async function nativePut(uid: string, blob: Blob): Promise<boolean> {
	try {
		await write_blob({ path: nativePath(uid), directory: NATIVE_DIR, blob, recursive: true });
	} catch {
		return false;
	}
	try {
		const base64 = await blobToBase64(blob);
		const { uri } = await MediaStoreSaver.saveToMusic({ fileName: nativeFileName(uid), base64 });
		if (uri) setStoredUri(uid, uri);
	} catch { /* public copy is visibility-only — best-effort */ }
	return true;
}
```

### WR-02: Whole-blob base64 round-trip across the Capacitor bridge — OOM/ANR risk the module's own header warns against

**File:** `src/lib/services/blob-store.ts:94-99, 106-107` (and `MediaStoreSaverPlugin.kt:63`)
**Issue:** The header explains step 1 uses `capacitor-blob-writer` precisely because base64 means "+33% bloat + a memory spike for large lossless files" — then step 2 does exactly that: `blobToBase64` builds a binary string one `String.fromCharCode` call per byte (a 60 MB FLAC = 60M iterations and a ~120 MB UTF-16 intermediate), `btoa` produces an ~80 MB string, the Capacitor bridge serializes it into a JSON message, and the Kotlin side materializes the full `ByteArray` before a single write. For lossless downloads this risks WebView OOM crashes / ANRs — and a crash here is not catchable by the never-throws try/catch.
**Fix:** Don't ship bytes over the bridge. The private copy is already on disk after step 1; pass its location instead and copy natively:
```ts
const { uri: fileUri } = await Filesystem.getUri({ path: nativePath(uid), directory: NATIVE_DIR });
await MediaStoreSaver.saveToMusic({ fileName: nativeFileName(uid), sourcePath: fileUri });
```
Kotlin side: open the source file and stream it into `resolver.openOutputStream(uri)` in chunks.

### WR-03: `nativeGet` decodes the entire file via base64 on every offline play

**File:** `src/lib/services/blob-store.ts:115-130`
**Issue:** `Filesystem.readFile` without `encoding` returns the whole file base64-encoded in one string; the code then `atob`s it byte-by-byte into a `Uint8Array` and wraps a Blob. For a large lossless file this is the same memory spike as WR-02 on every offline playback start (and on `restore()`).
**Fix:** On native, skip the Blob entirely: resolve the file's URI (`Filesystem.getUri`) and feed `Capacitor.convertFileSrc(uri)` straight to `<audio>.src`. The player already branches on native; the web IDB path keeps returning Blobs.

### WR-04: Public Music file name hardcodes `.mp3` regardless of actual format

**File:** `src/lib/services/blob-store.ts:64-66` (interacts with `MediaStoreSaverPlugin.kt:39-48`)
**Issue:** `nativeFileName()` always appends `.mp3`. A FLAC or M4A download (the lossless tiers QQ/Kuwo/JOOX return are common) lands in public `Music/OpenMusic/` named `*.mp3`, and the plugin's `mimeForFileName` then stamps `audio/mpeg` into MediaStore — wrong extension and wrong MIME for other players/file managers, the very consumers the public copy exists for. The web download path in TrackMenu already extracts the real extension from the URL, so the signal exists.
**Fix:** Thread the real extension into `put()` (or derive it in `nativePut` from `blob.type` / the resolved `audioUrl`), e.g. `nativeFileName(uid, ext)` defaulting to `mp3` only as a last resort.

### WR-05: IS_PENDING orphan when the stream write throws — pending MediaStore row never deleted

**File:** `android/app/src/main/java/com/openmusic/app/MediaStoreSaverPlugin.kt:82-91, 110-112`
**Issue:** The null-stream branch correctly deletes the inserted row, but if `it.write(bytes)` throws mid-write (disk full, I/O error) the exception unwinds to the outer catch (line 110), which rejects WITHOUT `resolver.delete(uri, ...)`. The row stays `IS_PENDING=1` (invisible, reclaimed by the system only after ~a week) and a partial file lingers; the TS side recorded no URI, so `del()` can never clean it.
**Fix:** Track the inserted uri and clean it on any failure after insert:
```kotlin
var pendingUri: Uri? = null
try {
    ...
    pendingUri = resolver.insert(collection, values) ?: ...
    ...
} catch (e: Exception) {
    pendingUri?.let { resolver.delete(it, null, null) }
    call.reject(e.message ?: "saveToMusic failed")
}
```

### WR-06: Plugin trusts caller-supplied paths — `fileName` traversal on the legacy branch; `deleteFromMusic` deletes arbitrary `file://` paths

**File:** `android/app/src/main/java/com/openmusic/app/MediaStoreSaverPlugin.kt:99, 130-137`
**Issue:** The plugin is callable by any JS in the WebView. On the API<=28 branch, `File(targetDir, fileName)` with `fileName = "../../<x>"` escapes `Music/OpenMusic/` into any public-storage location; `deleteFromMusic`'s `file` branch does `File(uri.path).delete()` on whatever path it is handed (including app-internal files the process can write). The current TS caller sanitizes (`blob-store.ts:54-65`), so this is defense-in-depth — but the sanitization lives on the wrong side of the trust boundary for a native-bridge method.
**Fix:** In `saveToMusic`, reject names containing separators: `if (fileName.contains('/') || fileName.contains('\\') || fileName == "..") { call.reject("invalid fileName"); return }` — and in `deleteFromMusic`'s `file` branch, verify the canonical path starts with the `Music/OpenMusic` directory's canonical path before deleting.

### WR-07: `versionCode = GITHUB_RUN_NUMBER` resets if the workflow is renamed/recreated — permanent downgrade block for installed users

**File:** `.github/workflows/android-release.yml:55`
**Issue:** `GITHUB_RUN_NUMBER` is scoped to the workflow *file*: renaming `android-release.yml`, moving it, or recreating the repo resets it to 1. The next release then carries a lower `versionCode` than what users have installed, and Android refuses the update — exactly the "Pitfall 5 — never downgrade-blocked" failure the comment claims to prevent, made permanent until someone hand-edits version logic. Re-running a failed run also reuses the same number (re-tagging produces a duplicate-versionCode release).
**Fix:** Derive versionCode from something monotonic and rename-proof, e.g. from the tag itself (`major*10000 + minor*100 + patch`) or `git rev-list --count HEAD`, or at minimum add a large fixed offset (`$((GITHUB_RUN_NUMBER + 10000))`) and document the invariant.

### WR-08: Signing keystore + passwords handed to an archived third-party action pinned to a mutable tag

**File:** `.github/workflows/android-release.yml:68-74` (also lines 32, 34, 38, 80)
**Issue:** `r0adkll/sign-android-release` is unmaintained/archived, and `@v1` is a mutable tag — if that tag is ever moved (account compromise, takeover), the job hands the attacker the release keystore, both passwords, and the alias, i.e. the Obtainium update trust anchor for every installed user. Same mutable-tag exposure applies (with lower stakes) to the checkout/setup/gh-release actions.
**Fix:** Pin all third-party actions to full commit SHAs (`uses: r0adkll/sign-android-release@<40-char-sha>`), or eliminate the dependency: decode the keystore to a temp file and run `zipalign` + `apksigner sign --ks ... --ks-pass env:KEYSTORE_PASSWORD ...` directly (both ship on the runner image; env-var password passing keeps it masked and out of `ps`).

### WR-09: `lastfm.ts`/`similar.ts` cache transient failures for 6h — contradicts the WR-03 cache posture the sibling module documents

**File:** `src/lib/services/lastfm.ts:56-69`, `src/lib/services/similar.ts:32-44`
**Issue:** `deezer.ts` documents and implements the rule that failures must REJECT inside `cached()` so they are never stored ("a transient failure ... is never pinned for the TTL"). `lastfm.ts`'s `fetchInfo` does the opposite: the try/catch lives INSIDE the cached factory and returns `{}` on any network error — pinning an all-empty enrichment for 6 hours after one timeout. It also never checks `res.ok`, so a 500/HTML-error JSON parse failure takes the same pinned-empty path (or caches a garbage shape if it parses). `similar.ts` has the same pattern: a transient `/api/similar` failure falls through to Deezer, and a Deezer `[]` failure-sentinel is then cached for 6h, hiding the similar-artists feature for the session. On native this matters more: a flaky mobile link at app start pins empties across the listening session.
**Fix:** Mirror the deezer posture — throw inside the factory, map outside:
```ts
return cached(`lf:info:${qs}`, TTL_LASTFM, async () => {
	const res = await apiFetch(`/api/lastfm/info?${qs}`);
	if (!res.ok) throw new Error(String(res.status));
	return ((await res.json()) as LastfmInfo) ?? {};
}).catch(() => ({}) as LastfmInfo);
```
(and equivalently in `getSimilarArtists`, only falling back to Deezer on a *successful-but-empty* Last.fm answer or outside the cache).

### WR-10: Kotlin jvmTarget unpinned while Java compatibility is fixed at 21 — jvm-target mismatch build failures depending on the build JVM

**File:** `android/build.gradle:37-39` (with `android/app/capacitor.build.gradle:4-7`)
**Issue:** The root script applies `org.jetbrains.kotlin.android` to `:app`, but no `kotlinOptions.jvmTarget`/toolchain is configured anywhere. `capacitor.build.gradle` pins Java source/target to 21. KGP's default JVM target follows the JVM running Gradle, so the build only succeeds when the local/CI JDK happens to be 21 (CI sets it; a contributor on JDK 17 gets "Inconsistent JVM-target compatibility ... 'compileReleaseJavaWithJavac' (21) and 'compileReleaseKotlin' (17)"). The capacitor file is regenerated by `cap update` ("DO NOT EDIT"), so the pin must live with the Kotlin plugin application.
**Fix:** In the same `project(':app')` block:
```groovy
project(':app') {
    apply plugin: 'org.jetbrains.kotlin.android'
    tasks.withType(org.jetbrains.kotlin.gradle.tasks.KotlinCompile).configureEach {
        kotlinOptions { jvmTarget = '21' }
    }
}
```

## Info

### IN-01: VITE_API_BASE documentation drift — `openmusic.lol` vs `openmusic.pages.dev`

**File:** `.github/workflows/android-release.yml:47`, `src/lib/services/api-base.ts:9`, `src/lib/proxy/http.ts:11-12`
**Issue:** The workflow bakes `VITE_API_BASE=https://openmusic.lol` while `api-base.ts` documents "the deployed Pages origin" / `https://openmusic.pages.dev`, and the CORS allowlist contains only `*.pages.dev` + localhost patterns. Native still works (the WebView origin `https://localhost` is allowlisted regardless of API host), but the docs and allowlist should name the real production host — and if `openmusic.lol` ever serves a page that cross-origin-calls the pages.dev deployment (or vice versa), the allowlist will need the missing origin.
**Fix:** Align the api-base.ts comment with the actual host; add `/^https:\/\/openmusic\.lol$/` to the allowlist if any cross-host browser call is possible.

### IN-02: Stale uid-shape comment and inaccurate "collision-free" claim in blob-store

**File:** `src/lib/services/blob-store.ts:53-55`
**Issue:** The comment says uids are hyphen-form (`netease-123`) but every adapter emits colon-form (`netease:123`, D-10). The `replace(/[^a-zA-Z0-9._-]/g, '_')` sanitize is also not strictly "collision-free": `a:b` and `a_b` map to the same path (harmless today because source ids contain no `_`-ambiguous separators, but the claim overstates).
**Fix:** Update the comment; if true collision-freedom is wanted, encode instead of replace (e.g. `encodeURIComponent` then replace `%` with `_25`-style escapes).

### IN-03: Hook overwrites `Vary` and preflight lacks `Access-Control-Max-Age`

**File:** `src/hooks.server.ts:34-36`
**Issue:** `response.headers.set('Vary', 'Origin')` clobbers any `Vary` value a route set (none today, but the seam is global). The 204 preflight also carries no `Access-Control-Max-Age`, so the native app pays a preflight round-trip per translate call.
**Fix:** Use `headers.append` for `Vary` (dedupe-aware) and add `'Access-Control-Max-Age': '86400'` to `corsHeaders`.

### IN-04: Translate endpoint: unbounded input → oversized upstream GET; cache-control on a POST is ineffective

**File:** `src/routes/api/translate/+server.ts:45-53, 34`
**Issue:** `lines` has no count/length cap; a full lyric set is joined and URL-encoded into a single GET query string, which can exceed upstream URL limits (414 → silent originals) and burns edge CPU on arbitrarily large bodies. The `cache-control: public, max-age=86400` header on a POST response is not cached by browsers/CDNs.
**Fix:** Cap input (e.g. `lines.length <= 200` and total chars <= ~10k, else return originals); drop the misleading cache header or split very long inputs into batched requests.

### IN-05: `android:allowBackup="true"` backs up WebView app data

**File:** `android/app/src/main/AndroidManifest.xml:5`
**Issue:** Auto-backup includes WebView localStorage/IndexedDB (library, player state, blob index). Nothing secret lives there today, but restored-state drift across devices is a known source of weird bug reports, and the surface grows as the app does.
**Fix:** Either set `android:allowBackup="false"` or add `android:dataExtractionRules` scoping what is backed up.

### IN-06: Repeated `put()` without `del()` duplicates public Music entries and orphans the previous URI

**File:** `src/lib/services/blob-store.ts:101-113`
**Issue:** `saveToMusic` always inserts a new MediaStore row; inserting the same DISPLAY_NAME again makes Android auto-rename (`x (1).mp3`), and `setStoredUri` overwrites the old content URI — the prior public copy becomes undeletable via `del()`. Triggerable by re-downloading a track at a different quality without removing it first.
**Fix:** In `nativePut`, call `MediaStoreSaver.deleteFromMusic({ uri })` for any existing stored URI before saving the new copy (or have the plugin upsert by DISPLAY_NAME+RELATIVE_PATH query).
