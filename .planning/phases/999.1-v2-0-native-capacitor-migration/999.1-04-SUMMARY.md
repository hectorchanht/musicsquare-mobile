# Phase 999.1 Plan 04: Native Media-Session Bridge + Background Audio Summary

Bridged the existing Web Media Session seam to a native Capacitor plugin so playback survives
screen-lock with a system media notification and working lock-screen transport (D-04/D-05),
without ever reloading the WebView on resume (D-06). The player's single `ms` accessor now branches
on `Capacitor.isNativePlatform()`: native routes to a thin adapter over
`@jofr/capacitor-media-session@4.0.0` (Cap-8 compatibility confirmed by the Task-1 spike);
web keeps `navigator.mediaSession` byte-for-byte. The pure media-session.ts helpers are untouched,
the full ~760-test suite stays green, and `assembleDebug` builds the APK clean under Cap 8.4.0 /
AGP 8.13 / JDK 21. **Task 3 (the device test matrix) is a blocking human-verify gate and has NOT
been signed off** — no physical device is currently attached via adb, and the 5+-minute locked-playback
/ lock-screen-transport / no-reload-resume checks require a human-held device.

**Task 2 — Native media-session bridge + FGS manifest + resume-safe lifecycle (D-04/D-05/D-06)** [commit fc0e706]
- Installed and pinned `@jofr/capacitor-media-session@4.0.0`. `@capacitor/app` already present (Plan 02).
- New `src/lib/services/native-media-session.ts`: `createNativeMediaSession()` returns a
  `PlayerMediaSession` adapter whose `metadata`/`playbackState` setters, `setPositionState(state?)`,
  and `setActionHandler(action, cb)` mirror the Web surface the player already calls and forward each
  to the plugin's `setMetadata`/`setPlaybackState`/`setPositionState`/`setActionHandler`. Plugin calls
  are fire-and-forget with a swallowing `.catch` (the Web surface is synchronous and the player must
  never block/throw on a UI update). Getters return the last-written value so read-after-write idioms
  stay consistent. `PlayerMediaSession` is the minimal structural interface both `navigator.mediaSession`
  and the adapter satisfy — so the `ms` accessor return type widened to it with zero call-site changes.
- `player.svelte.ts`: the `ms` accessor (formerly returning `navigator.mediaSession`) now branches —
  `if (Capacitor.isNativePlatform())` returns a lazily-built cached native adapter; otherwise returns
  `navigator.mediaSession` exactly as before (SSR + feature-detection guard preserved). The two things
  the plugin requires were ALREADY in place and now take effect natively: explicit
  `ms.playbackState = 'playing'` on every metadata write, and all transport action handlers
  (play/pause/previoustrack/nexttrack + seek) registered once in `attach()`. The FRESH-MediaMetadata
  idiom and `buildArtwork(this.resolvedCover)` are reused verbatim. The pure helpers in
  media-session.ts are byte-unchanged (`git diff --stat` empty).
- `AndroidManifest.xml`: added `FOREGROUND_SERVICE` + `FOREGROUND_SERVICE_MEDIA_PLAYBACK`. The
  jofr plugin merges in its `MediaSessionService` with `android:foregroundServiceType="mediaPlayback"`,
  but declares only `FOREGROUND_SERVICE` (stale, pre-Android-14) — so the media-playback permission is
  added here (Rule 2) to avoid `MissingForegroundServiceTypeException` on Android 14+/targetSdk 36.
- `npx cap sync android` registered jofr in the native project; it also picked up
  `@capacitor/filesystem` and `capacitor-blob-writer` (added by Plan 03 but never cap-synced into
  `android/`) — `capacitor.build.gradle` + `capacitor.settings.gradle` now list all 6 plugins.
- D-06: no `App.addListener('resume'|'appUrlOpen', …)` handler was added; no `invalidateAll()` /
  navigation anywhere. The `attach()` `visibilitychange` flush (persists position, does not reload)
  is unchanged. Capacitor restores the existing WebView on resume automatically.

## Deviations from Plan

### [Rule 2 — Missing critical functionality] FOREGROUND_SERVICE_MEDIA_PLAYBACK permission
- **Found during:** Task 2 (inspecting the jofr plugin's bundled AndroidManifest).
- **Issue:** The plugin declares only `FOREGROUND_SERVICE`, not the Android-14+/targetSdk-36-REQUIRED
  `FOREGROUND_SERVICE_MEDIA_PLAYBACK`. Starting the FGS without it throws
  `MissingForegroundServiceTypeException` (the exact D-04 failure). The plan's interfaces block called
  for adding it; confirmed necessary by the stale plugin's omission.
- **Fix:** Added both FGS permissions to the app manifest; verified the media-playback permission and
  the `mediaPlayback` service type both land in the merged manifest.
- **Files:** android/app/src/main/AndroidManifest.xml — **Commit:** fc0e706

### [Rule 3 — Blocking issue] Plan-03 plugins never cap-synced into android/
- **Found during:** Task 2 `npx cap sync android`.
- **Issue:** Plan 03 added `@capacitor/filesystem` + `capacitor-blob-writer` to package.json but never
  ran `cap sync`, so they were absent from `capacitor.build.gradle`/`capacitor.settings.gradle`. The
  required cap sync for jofr necessarily registered them too.
- **Fix:** Committed the regenerated gradle files (now list all 6 plugins). No source change.
- **Files:** android/app/capacitor.build.gradle, android/capacitor.settings.gradle — **Commit:** fc0e706

No bugs (Rule 1) in the bridge. No architectural (Rule 4) changes. No auth gates. jofr was the
gate-approved package, so no new package-legitimacy checkpoint was required.
