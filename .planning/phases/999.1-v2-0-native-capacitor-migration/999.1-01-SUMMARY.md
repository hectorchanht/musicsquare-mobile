# Phase 999.1 Plan 01: API_BASE Fetch Seam + Capacitor CORS Summary

Introduced the `apiUrl()/apiFetch()` fetch seam (D-03) and a single `hooks.server.ts` CORS
seam for every `/api/*` route (D-02), making the web codebase ready for the native APK to
call the deployed Pages proxy cross-origin — while leaving the web build byte-identical
(no env var set → same-origin relative fetch, web CORS unchanged).

## What Was Built

**Task 1 — API_BASE fetch seam + migration (D-03, Pitfall 3):**
- `src/lib/services/api-base.ts`: `apiUrl(path)` returns `BASE + path` where
  `BASE = import.meta.env.VITE_API_BASE ?? ''` (read lazily per-call so `vi.stubEnv`
  flips behavior in tests); `apiFetch(path, init)` funnels through `apiUrl` then calls
  global `fetch` exactly once.
- `api-base.test.ts` pins all four behaviors: empty-base→relative, set-base→absolute,
  single-fetch funnel with passed init, and the **Pitfall-3 proof** that a resolved
  Netease `track.audioUrl`/`lrcUrl` is absolute when a base is set (these flow straight
  into `<audio>.src`, so a relative URL would 404 in the APK).
- Migrated every relative `/api/*` call site to the seam: netease (incl. the audio/lrc
  URL assignments — the Pitfall-3 fix), qq, kuwo, joox, fivesing, jamendo, lastfm,
  translate, similar, and deezer (6 fetch sites). Each module's error contract
  (throw-on-drift / swallow-lyrics) is preserved verbatim — only URL construction changed.

**Task 2 — Capacitor CORS via single hook (D-02):**
- Added `/^https:\/\/localhost$/` (Capacitor Android default WebView origin) and
  `/^capacitor:\/\/localhost$/` (future iOS) to `ALLOWED_ORIGIN_PATTERNS`. Posture kept:
  echo-only-if-allowlisted, `Vary: Origin`, NEVER `Access-Control-Allow-Origin: *`.
- `src/hooks.server.ts` `handle()`: for `/api/*` requests, OPTIONS returns 204 without
  calling `resolve()` (workerd preflight correctness); non-OPTIONS resolves then merges
  `corsHeaders(origin)` onto the response. Non-`/api/*` paths pass through untouched.
  This single seam fixes `/api/translate` (which had no CORS) and every other route at once.
- Updated the translate route's doc comment to reflect central CORS handling.
- `proxy.test.ts`: 5 new hook tests — allowlisted-origin echo (incl. `https://localhost`),
  unknown-origin omission, OPTIONS 204 without resolve, never-`*`, and non-`/api/*` passthrough.

## Deviations from Plan

**Scope clarification (not a behavior change):** The plan's `files_modified` listed
`itunes-cover.ts`, `cover-backfill.ts`, and `discovery.ts`, but none has a direct `/api/*`
fetch to migrate — `itunes-cover.ts` calls `https://itunes.apple.com/search` directly
(external, CORS-open, not own-origin), and the other two delegate to the deezer/lastfm
helpers (which WERE migrated). They were left unchanged. The `<interfaces>` note already
flagged itunes-cover only as a "path-const" file; there was no `/api/*` surface there.
All other listed files were migrated as specified.

No bugs, missing functionality, or blockers encountered. No auth gates.

## Threat Surface

No new threat surface introduced beyond the threat model. The two added allowlist patterns
narrow exactly to the Capacitor WebView origins (T-999.1-01), the central hook closes the
translate-route CORS gap (T-999.1-02), and OPTIONS 204-without-resolve satisfies the
workerd preflight requirement (T-999.1-03). The open-relay invariant (never `*`) is
asserted by test.
