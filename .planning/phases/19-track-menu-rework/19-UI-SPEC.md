## Spacing Scale

The codebase uses an effective 4 / 8 / 12 / 16 / 18 / 24 rhythm. Declared as multiples of 4; the real values in use in TrackMenu / NowPlaying are noted per token. **12px is a genuine in-system step** (`.mi` row padding + `.mi` icon→label gap) — it is a multiple of 4 and must be preserved, not "rounded" to 8 or 16.

| Token | Value | Usage (real, in-system) |
|-------|-------|-------------------------|
| xs | 4px | tightest gaps; `--radius` halving; `.sk-ico` corner |
| sm | 8px | sheet outer padding (`.menu` `padding: 8px`); skeleton internal rhythm; `.menu-head` vertical padding |
| (step) | 12px | **`.mi` row padding** (`padding: 12px`) and **`.mi` icon→label gap** (`gap: 12px`); sheet horizontal inset (`left/right: 12px`) — the dominant row metric, keep verbatim |
| md | 16px | sheet bottom offset (`bottom: 16px`); legacy radius |
| (step) | 18px | header top-right icon cluster gap target; `--radius-lg` |
| lg | 24px | section-level breathing room (not heavily used inside the sheet) |

Sheet container metrics (from `.menu` / `.modal`, keep as-is): `left: 12px; right: 12px; bottom: 16px; padding: 8px; border-radius: 16px; max-width: 680px; margin: 0 auto; max-height: 80vh; overflow-y: auto`.

Exceptions:
- **44px minimum touch target** for the header top-right Like and Close icon buttons (icon glyph 18–20px inside a ≥44×44 hit area) — accessibility minimum, overrides the visual icon size.
- The `.mi` row's 12px padding + 18px icon makes each row ~44px tall already; preserve that height so the sheet does not jump between skeleton and resolved states.

## Typography

Sizes are scaled at runtime by the user's `--fs-*` multipliers (`settings.applyTheme()` sets these on `<html>`); base values below are the `1×` defaults. Declared: **4 sizes, 2 weights** (regular 400 / semibold 600 — bold 700+ is reserved for the NowPlaying big title only and is out of scope for the menu sheet).

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Header — song name (row 1) | 15px × `--fs-title` | 600 (semibold) | 1.25 |
| Header — artist name (row 2) | 13px × `--fs-artist` | 400 (regular) | 1.25 |
| Action row label (`.mi`) | 15px | 400 (regular) | normal |
| Meta / count / muted label (`.menu-head` legacy, `.count`, detail `dt`) | 12–13px × `--fs-title` where applicable | 400 (regular) | normal |

Notes:
- The **old single header line** (`.menu-head`, 13px muted, `{title} · {artist}` ellipsis) is REPLACED by the two-row marquee header below. Row 1 (song) is the visually dominant line: bump from the old muted 13px to **15px / weight 600 / `--color-text`**. Row 2 (artist) stays **13px / weight 400 / `--color-text-muted`** scaled by `--fs-artist`.
- Use `--fs-title` on the song row and `--fs-artist` on the artist row to stay consistent with how list rows (`.r-title` / `.r-artist`) and NowPlaying scale; do NOT use the NP-only `--fs-np-*` multipliers here (those are tuned for the ~1.5rem NP title and would oversize a sheet header).
- Tabular numerals are not needed in this sheet (no countdown lives here — the sleep-timer readout lives in the nowbar / NowPlaying).

**Accent (`--color-primary`) reserved for, in THIS phase, ONLY:**
1. The **Like (heart) active/liked state** in the header top-right cluster (filled heart, `fill="currentColor"` + `color: var(--color-primary)`), mirroring NowPlaying's `.t.on` transport-heart idiom.
2. The **`.mi.accent` primary affordance** that already exists (New Playlist `+` in the playlist sub-sheet) — unchanged.
3. **Focus rings** on the focusable header buttons (Like, Close) and `.mi` rows (`:focus-visible` outline color).

Accent is NOT used for: ordinary `.mi` rows, the Remix row label (it is a normal text row — see interaction contract), the inline resolving spinner (spinner uses `--color-text-muted`, NOT accent — it is a neutral "working" affordance, not a brand moment), the Close (X) glyph (uses `--color-text`), or row hover backgrounds (those use `--color-surface`).

Destructive: no new destructive color. The only destructive-adjacent action in the sheet is **Clear queue** (`Trash2`), which is pre-existing, gated to `queue.length > 1`, non-confirmed (a single tap, recoverable by re-generating), and stays a normal `--color-text` row — do NOT promote it to a red/destructive treatment in this phase (out of scope; would be a behavior change).

## Phase-Specific Visual & Interaction Contract

> **This is the load-bearing output of the spec.** The four requirements map to four contracts below. The executor implements `src/lib/components/TrackMenu.svelte` (contracts 1–3) and the long-press TRIGGER SITES (contract 4) against these.

### Overlay invariant (MUST NOT BREAK — applies to all contracts below)

The TrackMenu `$effect` that registers with the `overlays` store must keep its **dep as `open` ONLY** (never `track`), with `untrack()` around `overlays.open/dismiss`, the cleanup as the SOLE `dismiss` caller, and `{#if open && track}` as the visibility gate. The stub→resolved `track` reassignment (home long-press) must NOT re-run that effect (re-running churns history depth → over-pops Back to the previous route). The new Close (X) affordance (contract 1) MUST converge on this single dismiss path — it only flips `open`/`onclose()` false, exactly like the scrim and the existing detail-modal X; it never calls `overlays.dismiss` directly. (STATE.md Phase-19 blocker; CONTEXT D-09.)

### Contract 1 — Two-row marquee header with top-right Like + Close (MENU-02, D-08/D-09/D-10)

**Replaces** today's single `.menu-head` line (`{title} · {artist}`, muted 13px, `white-space:nowrap; text-overflow:ellipsis`).

**Layout** — a header region (`.menu-head` reworked, or a new `.sheet-head`) that is a flex row: a left text column (`flex: 1; min-width: 0`) and a right icon cluster (`flex: 0 0 auto`).

```
┌─────────────────────────────────────────────┐
│  ░░ Song name marquee row ░░          ♥  ✕  │   ← row1 song + top-right cluster
│  ░ Artist name marquee row ░               │   ← row2 artist (left column only)
└─────────────────────────────────────────────┘
```

- **Left text column** (display-only, NOT tappable — D-10): two stacked marquee clips.
  - Row 1: song name. Clip element gets `use:marquee`; inside it a `<span class="marquee-inner">{names.dnTitle(track.title)}</span>`. Clip is `overflow: hidden; white-space: nowrap; text-overflow: ellipsis; min-width: 0; max-width: 100%`. Typography per the Typography table (15px × `--fs-title`, weight 600, `--color-text`).
  - Row 2: artist name. Same marquee structure with `{names.dnArtist(track.artist)}` (13px × `--fs-artist`, weight 400, `--color-text-muted`).
  - **Copy NowPlaying's exact marquee structure** (`NowPlaying.svelte` lines 697–700): each row is `<div class="…" use:marquee><span class="marquee-inner">…</span></div>`. The marquee keyframe is GLOBAL in `app.css` (`@keyframes marquee-scroll`, `.marquee-on .marquee-inner`) — do NOT redefine it in the component; `use:marquee` only sets `--marquee-dx` + toggles `.marquee-on` on the clip. Reduced-motion → the action never adds `.marquee-on`, so the clip's `text-overflow: ellipsis` shows a static ellipsis (no animation). This satisfies the project marquee rule (parent locked `flex` / `min-width:0` / `max-width`; `marquee-inner` child).
  - **Marquee re-measure:** wrap the two rows in `{#key track.uid}` so a stub→resolved `track` reassignment (text width change) remounts the clips and forces `use:marquee` to re-measure — mirror NowPlaying's `{#key player.current?.uid}`. (ROADMAP "marquee re-measure" flag.)
- **Right icon cluster** (`.head-actions`): Like then Close, side by side, vertically centered against the header, `gap: 18px` between the two hit areas (or 12–18px; ≥12px so the two 44px targets do not overlap).
  - **Like (heart):** `Heart` from `@lucide/svelte`, glyph `size={20}` inside a ≥44×44 button. Idle = `fill="none"`, `color: var(--color-text)` (or muted, matching the row default). Liked = `fill="currentColor"`, `color: var(--color-primary)` (the ONLY accent in the header). `aria-label` = `menu.like` (idle) / `menu.liked` (active); add `aria-pressed={liked}`. Tapping calls the existing `like()` (toggles + toasts). This MOVES Like out of the mid-list — **remove the mid-list `menu.like` `.mi` row** (D-09) to avoid duplication.
  - **Close (X):** `X` from `@lucide/svelte`, glyph `size={20}` inside a ≥44×44 button, `color: var(--color-text)`. `aria-label` = `menu.closeMenu` (reuse existing). Tapping calls `close()` (flips state false → single dismiss path; D-09). This is a NEW explicit affordance (today close is scrim/drag only) and matches the existing detail-modal X (TrackMenu line 218).
- Like is on the LEFT of the cluster, Close on the RIGHT (closest to the screen edge), matching the Spotify / YT-Music sheet-header convention.

**State coverage:** liked/unliked (heart fill + accent), focus-visible ring (accent) on both icon buttons, `:active`/press feedback consistent with `.icon:hover { background: var(--color-surface-2) }` (NowPlaying header idiom) — give the header buttons a subtle hover/active background using `--color-surface`/`--color-surface-2`, NOT accent.

### Contract 2 — Always-visible buttons + gated resolve-then-act with inline spinner (MENU-01, D-01/D-02/D-03/D-11)

**Replaces** today's `{#if loading} 9× .mi-skel rows {:else} buttons {/if}` gate, which hides EVERY button behind a skeleton until the stub resolves.

**New behavior:** ALL action rows render immediately the moment the sheet opens — even on an unresolved discovery stub (home long-press). The opening **skeleton is reduced to the HEADER region only** (the two stacked header bars, contract 1's shape), covering the brief pre-data / marquee-remeasure window; the action list is NEVER skeletoned.

- **Header skeleton (D-11) — matches the new 2-row shape, NOT the old 9-row list:** while `loading` and the header text is not yet present, show **two stacked `.sk` bars** sized to the title/artist rows:
  - Bar 1 (song): height ~15px (matches row-1 line box), width ~60–70% of the text column.
  - Bar 2 (artist): height ~12px, width ~40–50%, with ~6px vertical gap below bar 1.
  - Use the GLOBAL `.sk` class from `app.css` (`background: rgba(255,255,255,0.11)` + shimmer; reduced-motion drops the shimmer, keeps the grey block) — do NOT reintroduce the bespoke `.sk-bar`/`.mi-skel`/`mi-shimmer` styles; those skeleton rows are removed with the old gate. Right-cluster (Like/Close) renders real (not skeletoned) since it works on the stub.
  - Home stubs already carry title/artist, so in practice the header text is usually present immediately; the skeleton only fills the rare pre-data instant and prevents layout jump (project skeleton rule: match shape/size of the loaded content).
- **Ungated actions (work on the stub object alone — render + active immediately):** Play next, Add to queue, Like (now in header), Add to playlist, Go to artist, Sleep timer, Share, plus Shuffle queue / Clear queue (gated only on `queue.length > 1` as today, NOT on track resolution). These need only stub fields (`uid`, `title`, `artist`, `album`) the store/library already accept.
- **Gated actions (`detailsLoaded && uid` — need resolved `audioUrl`/details):** **Download**, **Detail**, **Remix-play** (contract 3). They render IMMEDIATELY and are tappable on a stub (resolve-then-act, D-02): tapping kicks off `ensureTrackDetails(...)`, shows the inline spinner on THAT row, and the action **fires automatically once data arrives**.
  - Apply the rule per-action by the literal `detailsLoaded && uid` test. **Share is NOT gated** (Claude's discretion, D-42): `shareUrl(track)` builds from `uid`/`title`/`artist` + queue, not resolved audio — it works on the stub.

**Inline spinner visual (Claude's discretion, specified here):**
- A small spinner replacing (or immediately right of) the row's leading 18px Lucide icon, occupying the same 18px box so the row does not shift width. Implement as a 16px circular CSS spinner: `width:16px; height:16px; border:2px solid var(--color-text-muted); border-top-color: transparent; border-radius:50%; animation: spin 0.7s linear infinite`. Color = `--color-text-muted` (neutral "working", NOT accent — see Color section).
- **Reduced-motion fallback:** under `prefers-reduced-motion: reduce` (and the app's `:root[data-reduce-motion]` rule, which already kills `animation` app-wide), the spinner must not rely on rotation alone. Fallback = the row label swaps to `menu.preparing` (`Preparing…`) text and/or the icon dims to a static partial-ring; the row is visibly "busy" without motion. Provide `aria-busy="true"` on the row and an `aria-label`/visually-hidden text of `menu.preparing` so screen readers announce it.
- **Disabled vs spinning, three distinct visual states:**
  1. **Idle/ready** (resolved, or ungated): full opacity (1.0), normal icon, label in `--color-text`, tappable.
  2. **Spinning/resolving** (gated action tapped, resolve in flight): leading icon replaced by the 16px muted spinner, `aria-busy="true"`, row still rendered at full label opacity (the spinner — not dimming — communicates "working"), `pointer-events` may stay on but a second tap is a no-op (see dedupe).
  3. **Disabled** (an action genuinely unavailable, e.g. `Clear queue` when `queue.length <= 1` is simply not rendered; `Go to album` precedent uses `disabled`): existing `.mi:disabled { opacity: 0.4; cursor: default; }`. Note: gated actions on a stub are NOT shown as disabled — they are tappable (resolve-then-act). "Disabled" is reserved for truly-unavailable actions, not for "data not yet resolved".
- **Double-tap dedupe (D-03):** exactly ONE resolve is in flight per action. While a row is spinning, a second tap is a no-op (guard with a per-action `inFlight` flag, e.g. a `Set<string>` of action keys or a per-row boolean). On resolve SUCCESS the queued action auto-runs once then the spinner clears; on resolve FAILURE the spinner clears and a graceful toast shows (`toast.noAudio` for audio-class failures) — **never a stuck spinner**.

### Contract 3 — Remix action (QUEUE-04, D-04/D-05/D-06/D-07)

A new `.mi` action row placed in the **queue-actions cluster, near Play next / Add to queue** (top of the action list, with the other queue-affecting rows — Play next, Add to queue, then Remix, then Shuffle/Clear when present). It is a normal text row (NOT accent-colored).

- **Icon:** `Sparkles` from `@lucide/svelte`, `size={18}` (matches all other `.mi` icons). Rationale: `Shuffle` is already taken by Shuffle queue and `Repeat`/`Repeat1` by the transport repeat control; `Sparkles` reads as "freshly generate / remix" and does not collide. (Confirmed present in `@lucide/svelte`.)
- **Label:** `menu.remix` → `Remix`.
- **Trigger feedback:** on tap, fire `toast.remixing` → `Remixing from this song` (D-07: toast on trigger).
- **Behavior (reuse existing path — do NOT build a new queue mechanism, D-04):** play the triggering track first via the existing fresh-play regenerate path — `player.play(track, { fresh: true })` → `regenerate(seed)` → `buildSimilarQueue`, which already yields `dedupeBest([seed, ...manualEntries, ...auto])`. This preserves user-pinned/manually-added tracks via `manualUids` and discards the prior generated tail (D-05).
- **Forced generation (D-06):** Remix must ALWAYS generate regardless of the user's per-context up-next setting (`'same-list'` vs `'generated'`). Force `effectiveUpnextMode === 'generated'` for the Remix play (e.g. set a Remix `queueContext` or otherwise override the gate). Exact mechanism is a planning/research detail — the UI contract only requires that an explicit Remix never falls back to same-list.
- **Gating (D-07):** Remix is a GATED action — it needs `audioUrl` to play the seed, so it gets the contract-2 resolve-then-act treatment: tappable on a stub, shows the inline row spinner while resolving, fires automatically on resolve, deduped against double-tap, graceful failure (`toast.noAudio`) on resolve failure.
- No sub-sheet is introduced for Remix (CONTEXT: "Remix needs no sub-sheet").

### Contract 4 — Long-press release: no stuck state + no double action (MENU-03, D-12)

This is an **interaction rule applied at the long-press TRIGGER SITES** (home tiles, track rows, compact rows — wherever `use:longpress onlongpress={() => openMenu(track)}` lives), **NOT inside TrackMenu**.

- **No stuck highlight:** after a long-press opens the menu, the trigger element under the finger must NOT retain a `:active` / `:focus` / `:hover` highlight while the sheet is open. Contract: the trigger row shows no residual pressed/focused/hovered styling once the menu is up. Mechanism is research/planning per iOS-Safari-vs-Android-Chrome behavior (candidates: explicitly `blur()` the trigger when `longpress` fires; emit/handle `pointercancel`; CSS `@media (hover: hover)` guards so touch never sticks `:hover`; `-webkit-tap-highlight-color: transparent`). The visible CONTRACT is: zero stuck highlight under the finger.
- **No double action (suppress the synthetic finger-up click):** the synthetic `click` that fires on finger-up after a long-press must be suppressed so the row does not ALSO play/navigate. The existing `src/lib/actions/longpress.ts` ALREADY implements this (`suppressNextClick`: a one-shot capture-phase click suppressor armed when the longpress timer fires, self-disarmed at ~700ms). Contract: every long-press trigger site uses `use:longpress` so this guard is in force; MENU-03's job is to verify/extend it covers the stuck-state half AND that it holds across the home tiles / compact rows / track rows on both iOS Safari and Android Chrome. Do not regress the trailing-click guard.
- **Where this lands:** the trigger components (home page tiles, search/library track rows, NowPlaying queue/related rows already wired with `use:longpress`), not `TrackMenu.svelte`. TrackMenu's only related responsibility is that opening on a stub (the home-tile path) must not break the overlay invariant (see top of this section).
