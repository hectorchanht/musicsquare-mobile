# Phase quick-260606-w87 Plan 01: Home Layout Customization (Settings Reorder/Hide, Subset, Density, Landing) — Summary

Added a `/settings/home` customization page and wired it into the home page + app shell: the user can drag-reorder and show/hide the four discovery sections, pick a subset of discovery tags/countries, set items-per-shelf (6–24), choose a default landing tab, toggle tile density, and hide the search pill / Randomize button. All config rides the existing `settings` runes store with a non-destructive localStorage migration, and **every default reproduces today's home exactly** — a returning user sees no change. A pure, unit-tested `home-layout.ts` module clamps/drops/falls-back on any corrupt or old persisted value so the home render can never be broken by bad localStorage. No new npm dependency.

## What Was Built

### Task 1 — `home-layout.ts` pure resolvers + settings fields (TDD)  ·  commit `681632d`
- **`src/lib/services/home-layout.ts`** — a pure leaf module (imports nothing; runs in the node Vitest project):
  - `HOME_SECTIONS = ['top-hits','top-artists','tags','countries']` + `DEFAULT_SECTION_ORDER` (a fresh copy).
  - `resolveSectionOrder(saved)` — undefined/empty/non-array → defaults; drops unknown/old ids; de-dupes; appends any missing known id in canonical order, so the result is ALWAYS a permutation-superset covering every section (a newly-shipped section never vanishes, a corrupt order never blanks the home — T-w87-02).
  - `resolveSubset(saved, pool)` — undefined/non-array → full pool; filters to pool members **in pool order** (not selection order); empty/all-invalid → full pool (the "fall back to full default" rule — T-w87-03); never mutates the pool.
  - `clampShelfSize(n: unknown)` — non-number/NaN/undefined → 18; floors fractionals; clamps to [6,24] (T-w87-01 — no giant/NaN/negative page size). `SHELF_MIN=6`, `SHELF_MAX=24`, `SHELF_DEFAULT=18`.
  - `HomeDensity`/`HomeLandingTab` types + `LANDING_PATHS` (fixed Record → no open-redirect, T-w87-05).
  - **Also OWNS the discovery pools** `DISCOVERY_TAGS` (8) / `DISCOVERY_COUNTRIES` (6) — relocated here (see Deviations).
- **`home-layout.test.ts`** — 24 cases: section-order resolution (keep/drop/append/de-dupe/full-permutation), subset fallback + pool-order + non-mutation, shelf clamp (large/small/valid/fractional/non-number/negative), LANDING_PATHS.
- **`settings.svelte.ts`** — 9 new `$state` fields with non-destructive load() migration (Array.isArray guard for arrays; `clampShelfSize(v.homeShelfSize)` on load; `homeLandingTab`/`homeDensity` via `?? default`; booleans default TRUE via `?? true`, NOT `!!`); all 9 added to save(). KEY unchanged, no version bump — a v1 blob with none of these fields loads with every default = today.

### Task 2 — `dragReorder` action + `/settings/home` page + group + i18n ×3  ·  commit `3b693e8`
- **`src/lib/actions/dragReorder.ts`** — vertical drag-to-REORDER action mirroring `dragScroll` (typed `Action`, returns `{update,destroy}`, no npm dep). Each row carries `data-reorder-index`; the grab handle carries `data-reorder-handle` + `touch-action:none` so a vertical drag reorders rather than scrolls the page. Captures the pointer, computes the drop index on release, fires `onReorder(from,to)`. Reduced-motion aware (lift only when `!settings.reduceMotion` and `!prefers-reduced-motion`). Pure `computeDropIndex(pointerY, rowTops, rowHeights, from)` extracted for tests.
- **`dragReorder.test.ts`** — 9 cases for the index math (no-op on own row, down→+1, up→span, far-down/far-up clamp, single-row, empty-geometry degrade).
- **`src/routes/(app)/settings/home/+page.svelte`** — 7 sections mirroring general/playback styles: (1) section reorder list (`use:dragReorder`, GripVertical handle) + per-row `.sw` visibility toggle iterating `resolveSectionOrder`; (2) genre-tag multiselect chips; (3) country multiselect chips, both with a "Showing all" hint when the selection is empty; (4) items-per-shelf `<input type=range min=6 max=24>` with the live value; (5) landing-tab `.seg`; (6) density `.seg`; (7) two chrome `.row-toggle`s. Every mutation calls `settings.save()`.
- **Group registered** in `/settings/+page.svelte` (LayoutGrid icon, right after General).
- **24 i18n keys** appended in a commented block to `en.ts`, `zh-Hant.ts`, `zh-Hans.ts` (parity test enforces identical key sets).

## Robustness / Threat Mitigations Implemented

- **T-w87-01** (poisoned shelf size): `clampShelfSize` floors+clamps to [6,24]; non-number → 18. Applied both on load() and in refresh(). No giant fan-out limit or NaN page size possible.
- **T-w87-02** (corrupt section order/hidden): `resolveSectionOrder` drops unknown ids, de-dupes, appends missing known ids; the render iterates the resolved order and intersects hidden against the known set. A corrupt order (e.g. `["bogus"]`) renders the full default order.
- **T-w87-03** (poisoned tags/countries): `resolveSubset` filters to the pool and falls back to the full pool when empty — a garbage tag is dropped before it reaches the edge; an empty/garbage selection never blanks discovery.
- **T-w87-04** (fan-out width): width is bounded by the POOL size and stays behind `FANOUT_CAP=4`; config can only NARROW the subset, never widen it — no new in-flight pressure.
- **T-w87-05** (poisoned landing tab): redirect target is read from the fixed `LANDING_PATHS` record, never the raw stored string; `?play=` deep links bypass the redirect.
- The corrupt-blob case from the checkpoint (`{"homeShelfSize":999,"homeSectionOrder":["bogus"]}`) is covered by the clamp/resolve unit tests (clamps to 24, renders the default order) and verified by the corrupt-value test cases.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Broke a circular import that crashed module load**
- **Found during:** Task 3 (full `pnpm test` run).
- **Issue:** The plan had `settings.svelte.ts` import `DISCOVERY_TAGS`/`DISCOVERY_COUNTRIES` from `discovery.ts`. But `discovery.ts` imports `settings` (for `resolveStub`), creating a `settings → discovery → settings` cycle. The `settings` singleton is constructed at module top-level, and its instance-field initializers spread `[...DISCOVERY_TAGS]` eagerly — when `discovery.ts` was the cycle entry (via `picks.ts`/`qq.ts`), `DISCOVERY_TAGS` was still `undefined`, throwing `TypeError: DISCOVERY_TAGS is not iterable` and failing two test suites (the player test's `vi.mock('$lib/services/discovery', () => ({ resolveStub }))` also lacked the constant). The plan's note that this edge was safe was incorrect for the eager top-level construction.
- **Fix:** Moved the `DISCOVERY_TAGS`/`DISCOVERY_COUNTRIES` constant definitions into the pure `home-layout.ts` (imports nothing); `discovery.ts` now `export { ... } from '$lib/services/home-layout'` so every existing consumer (and `discovery.test.ts`, the player mock) is unchanged; `settings.svelte.ts` imports the pools from `home-layout`. This removes the `settings → discovery` edge entirely — `settings → home-layout` is the only edge and `home-layout` imports nothing → no cycle.
- **Files modified:** `src/lib/services/home-layout.ts`, `src/lib/services/discovery.ts`, `src/lib/stores/settings.svelte.ts`.
- **Commit:** `a674873`.
- **Result:** 365/365 tests green; no consumer of `$lib/services/discovery` changed its import.

**2. [Rule 1 - Test correctness] Corrected one dragReorder test expectation**
- **Found during:** Task 2 (first GREEN run, 8/9).
- **Issue:** A test asserted `computeDropIndex(35, ...) === 1` from a "previous-row midpoint" mental model, but the implemented (and more intuitive) semantics are span-containment: y=35 sits inside row 0's span `[0,40)` → drop index 0.
- **Fix:** Replaced that case with two correct span-based assertions (y=50 in row-1 span → 1; y=35 in row-0 span → 0). Implementation unchanged.
- **Files modified:** `src/lib/actions/dragReorder.test.ts`. **Commit:** `3b693e8`.
