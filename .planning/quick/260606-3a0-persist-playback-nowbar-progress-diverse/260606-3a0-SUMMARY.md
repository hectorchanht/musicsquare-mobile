# Quick Task 260606-3a0 — Summary

Four UX upgrades to the deployed MusicSquare app. Live at openmusic.pages.dev.

## Shipped

1. **Playback persists across navigation.** Moved the single `<audio>` element + `player.attach()` to the **root** layout (`src/routes/+layout.svelte`) so it mounts once and is never torn down by client-side nav (home ↔ search ↔ artist ↔ /spike). The visible now-playing bar + overlay stay in `(app)/+layout.svelte` and read the same player singleton. Removed the duplicate `<audio>` from the (app) layout.
2. **Progress sliver on the mini nowbar.** A 3px violet fill pinned to the top edge of the mini bar, width = `currentTime/duration` (guarded; 0 until duration known). The full seekable bar remains in NowPlaying.
3. **Top picks = many hits from many distinct artists.** Home no longer searches one keyword. A pool of 20 artists (CN + western); `buildPicks()` samples 9 distinct artists, `Promise.allSettled(searchAll(artist))`, takes each artist's top result, dedupes by uid → a diverse grid (one artist per tile).
4. **localStorage cache + Randomize.** Picks cached under `musicsquare:top-picks:v1`; on mount the home renders instantly from cache (no network) and only fetches on a cold cache. The header **↻ Randomize** button samples a fresh artist set, refetches, overwrites the cache, and updates the grid (loading state shown). localStorage touched only in `onMount`/handlers (SSR-safe).

The picks list also seeds `player.setQueue(...)` so Up-Next reflects the diverse grid.

## Notes / scope

- Cache has no TTL — Randomize is the manual refresh (intentional: "don't reload every time"). A versioned key (`:v1`) lets future shape changes bust it.
- Cold-start Randomize fans out 9 × `searchAll` (each hits 4 sources) — fine for a cached one-time build; not rate-limited (Phase-7 concern).
- Reused Phase 1 data layer + existing player/catalog; no adapter/proxy edits.
