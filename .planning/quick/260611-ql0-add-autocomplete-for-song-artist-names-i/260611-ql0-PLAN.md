<objective>
Add typeahead autocomplete (live song + artist suggestions) to the search page. As the user
types, debounced suggestions appear below the search bar; tapping one fills the input and runs
the full multi-source search.

Purpose: Make search feel responsive and helpful — users discover the right query before
committing to a full fan-out search, without hammering any upstream API.

Output:
- A pure, node-testable autocomplete logic module (debounce primitive + suggestion derivation)
  with a vitest test, mirroring the existing `search/search-history-logic.ts` + `.test.ts` split.
- Suggestion UI + debounced/abort-guarded fetch wired into the existing search `+page.svelte`,
  coexisting with (never replacing) the recent-search history list.
- One new i18n key across all 15 locales (i18n parity test gate).

Source decision: **Deezer**, via the EXISTING client helper `deezerSearchTopN(term, limit, signal)`
in `src/lib/services/deezer.ts`. It is keyless (no new credential), CORS-proxied through the
already-shipped `/api/deezer/search` edge route (which clamps `limit` to [1,25], edge-caches 24h,
and never throws), already client-side TTL-cached, already honors an AbortSignal, and returns
`DeezerHit[]` with `{ title, artist, cover }`. NO new API route, NO new service fetch code, NO new
npm dependency, NO new env var. This is the cheapest fast source that already exists in the repo —
exactly matching the user's "you can see what source seem fit … cheap, fast, no new credentials".
(Last.fm `track.search`/`artist.search` would need a key that is NOT guaranteed wired client-side;
CN resolvers are slower/flakier and the search page already fans out to them on submit — using
them for typeahead would defeat the "not heavy to api fetching" goal. Pick ONE source for v1.)
</objective>
