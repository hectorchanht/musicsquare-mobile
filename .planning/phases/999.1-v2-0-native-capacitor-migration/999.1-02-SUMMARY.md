## Operational Findings (record for later waves)

- **Native APK verification REQUIRES the web deployment to be current.** During Task 3 the deployed Pages app was stale — the Plan-01 CORS hook was not live, so every cross-origin `/api/*` fetch was CORS-blocked. Orchestrator redeployed via `pnpm build && npx wrangler pages deploy .svelte-kit/cloudflare --project-name openmusic`; after redeploy, zero CORS errors. **Deploy is manual wrangler — there is no CI deploy.** Any future APK that relies on a new hook/seam must be preceded by a manual Pages deploy.
- **JDK 21 is mandatory** for `assembleDebug` (`JAVA_HOME=/opt/homebrew/opt/openjdk@21`); JDK 20 fails `invalid source release: 21`.
- **Samsung Secure Folder devices** need `adb install -r --user 0` (multiple Android users present).

## Known Issues (not fixed this plan)

- **Song loading noticeably slower in the APK than the web version.** Likely cross-origin `/api/*` latency from the WebView (https://localhost → https://openmusic.lol) plus no service-worker cache inside the WebView. To be addressed in a later wave (caching / SW shell). Not a regression — expected for the foreground-only slice.

## Threat Surface

No new threat surface beyond the plan's threat model. `allowMixedContent: false` + default `androidScheme: 'https'` keeps the WebView origin at `https://localhost` (T-999.1-04, allowlisted in Plan-01). The APK reuses the Plan-01 CORS allowlist cross-origin; device check confirmed only `https://localhost` is accepted (T-999.1-06). appId locked to `com.openmusic.app` before any signed release (T-999.1-05). The API-base change (openmusic.pages.dev → openmusic.lol) stays within the same trust boundary (same Pages project, same allowlist).
