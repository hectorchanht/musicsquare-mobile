FIX 1 — Flush half-offset + persistent resting half transform (BUG 1, both root causes).

1a. Add a transport element ref: declare `let transportEl = $state<HTMLElement | null>(null)` alongside the other sheet refs, and add `bind:this={transportEl}` to the existing `<div class="transport">` (line ~322).

1b. In `measureOffsets()` (line ~157), replace the arbitrary `halfOffset = Math.round(npRect.height * 0.5)` (line ~168) with a measurement of the live bottom edge of the transport row relative to `.np` top: when `transportEl` is available, `halfOffset = Math.round(transportEl.getBoundingClientRect().bottom - npRect.top)`; fall back to the old `npRect.height * 0.5` only if `transportEl` is null. Keep the existing sane clamp on the next line (`halfOffset = Math.max(20, Math.min(closedOffset - 20, halfOffset))`) so half always sits between full(0) and closed.

1c. Give the resting `half` state a persistent transform so it does not collapse to full coverage. The `.sheet` element's `style:transform` is currently `sheetDragging ? translateY(sheetDragY) : undefined` (line ~337). Drive it from BOTH drag and resting state: when dragging → `translateY(sheetDragY)`; else when `sheetState === 'half'` → `translateY(halfOffset)`; else `undefined`. (Resting full = no transform = inset:0 fills, correct; resting closed = sheet is in normal flow, transform undefined, correct.)

1d. Because the resting-half transform reads `halfOffset`, it must be MEASURED whenever the sheet enters `half` via a non-drag path (tap / keyboard) — `selectTab` and `gripKey` currently never call `measureOffsets()`. Ensure `halfOffset` is correct for resting half: add an `$effect` that, when `sheetState === 'half'` and not dragging, calls `measureOffsets()` (it is idempotent and already guards on null refs). This recomputes the flush offset on entering half from any path and on layout-affecting changes. Do NOT break the existing back-gesture `$effect` at lines ~120-123 — add a SEPARATE new effect; do not merge into it.

FIX 2 — Subnav item tap switches tab with priority over the generic grip toggle (BUG 2).

2a. In `gripDown` (line ~173), detect whether the gesture STARTED on a subnav button. Add a module-scope `let gripStartTab: Tab | null = null` (plain let, not $state — it is gesture-transient). Prefer the `data-tab` approach for robustness — add `data-tab="queue"|"lyrics"|"related"` to the three subnav buttons (lines ~348-350). Set `gripStartTab = btn ? (btn.dataset.tab as Tab) : null`.

2b. In `gripUp` (line ~191), modify ONLY the TAP branch (`|moved|<8`, lines ~194-200): if `gripStartTab` is non-null (the tap originated on a subnav button) → call `selectTab(gripStartTab)` and SKIP the generic `closed→half / full→half / half→closed` toggle (still reset `sheetDragging=false; sheetDragY=0; gripActive=false` as today and return). Otherwise (tap on the grip handle / empty nav area) → run the existing generic toggle unchanged. Reset `gripStartTab = null` at the end of every `gripUp` path (both tap and drag) so it never leaks into the next gesture.

2c. Confirm `selectTab` (line ~230) still guards `subnavMoved` (a drag never switches tabs) and remains idempotent: switching to the already-active tab plus `if (sheetState === 'closed') sheetState = 'half'` is safe to call twice (the buttons' native `onclick` may still fire as a fallback when pointer capture does not suppress it). Do not remove the buttons' `onclick={() => selectTab(...)}` — keep it as the fallback. Per the SUMMARY's single-dismiss-path discipline, do NOT touch the overlays/back-gesture `$effect`.
